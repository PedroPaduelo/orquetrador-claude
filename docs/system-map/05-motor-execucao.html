Caracteristicas

  • ▶ Output de cada step vira input do proximo
  • ▶ Conditions podem pular, repetir ou encerrar steps
  • ▶ Suporta retry com maxRetries configuravel
  • ▶ Validators rodam apos cada step completar

Metodo

executeSequential(context, userInput)

Itera pelo array de steps com while (i < steps.length), avancando baseado em conditions.

Como Funciona

  • ▶ DAGExecutor monta o grafo a partir de step.dependsOn
  • ▶ Valida ciclos com DFS antes de executar
  • ▶ getReadySteps() retorna steps com todas as deps concluidas
  • ▶ Contexto de dependencia injetado automaticamente

Concorrencia

Mecanismo de Pausa

  • ▶ Claude chama tool AskUserQuestion
  • ▶ Engine detecta e mata o processo (SIGTERM)
  • ▶ Estado salvo: markPaused(executionId, stepIndex, resumeToken)
  • ▶ Usuario responde, execucao retoma com --resume

User Interrupt

  • ▶ Usuario envia mensagem durante execucao ativa
  • ▶ Processo atual e morto (SIGTERM/SIGKILL)
  • ▶ Re-executa com --resume + mensagem do usuario
  • ▶ Loop while(true) garante continuidade

Integracao Claude CLI

Como o ClaudeCodeEngine invoca e gerencia processos do Claude

Argumentos da CLI

# Comando construido pelo ClaudeCodeEngine
claude --print --output-format stream-json --verbose --dangerously-skip-permissions
  # Opcionais:
  --resume <session-token>
  --system-prompt <prompt>
  --model <model-name>
  "mensagem do usuario"
--print

Modo nao-interativo. Output vai para stdout em vez do terminal interativo.

--output-format stream-json

Emite eventos JSON em tempo real: content, session, actions, errors. Parseado pelo StreamParser.

--dangerously-skip-permissions

Pula aprovacoes de tool use. Necessario para automacao sem interacao humana.

--resume <token>

Retoma sessao existente. Permite continuidade de conversa e interrupts do usuario.

--model <name>

Modelo especifico por step. Cada step pode usar um modelo diferente (opus, sonnet, haiku).

--verbose

Log detalhado no stderr. Capturado pelo monitor para debugging e observabilidade.

Gerenciamento de Contexto

Como o sistema mantem e compacta o contexto entre execucoes

System Prompt & Memoria

buildSystemPrompt({
  stepSystemPrompt: step.systemPrompt,
  projectPath: "/workspace/projeto",
  memory: stepMemory,
  useBasePrompt: true,
})
  • ▶ Base prompt + prompt customizado do step
  • ▶ Memory acumulada de compacoes anteriores
  • ▶ projectPath injetado como CWD do processo
  • ▶ Git config automatico (user.name, user.email)

Compactacao de Contexto

1 Prompt Too Long

Erro prompt_too_long detectado durante execucao com --resume

2 Sumarizacao

memoryManager.summarizeAndSave() usa claude-haiku-4-5 para resumir o contexto acumulado

3 Reset de Sessao

Sessao antiga deletada, nova sessao criada com memoria compactada no system prompt

4 Re-execucao

Step re-executa com resumeToken: null e novo systemPrompt contendo a memoria

Tratamento de Erros

Estrategias de recuperacao em cada cenario de falha

Retry com Conditions

  • • Conditions avaliam output de cada step
  • • maxRetries configuravel por step (padrao: 3)
  • • Validators podem forcar retry com feedback
  • • Jump backward com limite de repeticoes
retryCounts[step.id]++
if (retry >= maxRetries) skip

Timeout

  • • Padrao: 600 segundos por step
  • • SIGTERM enviado ao processo group
  • • SIGKILL apos 2s se nao encerrar
  • • timedOut: true no resultado
process.kill(-pid, 'SIGTERM')
setTimeout(() => kill(-pid, 'SIGKILL'), 2000)

Falhas de Sessao

  • • Session Not Found: deleta session, retenta sem --resume
  • • Prompt Too Long: compacta memoria, nova sessao
  • • Step Error: emite evento, marca execucao como failed
  • • Cancel: SIGTERM + cleanup de activeExecutions
isSessionNotFoundError(err)
isPromptTooLongError(err)

Ciclo de Vida da Execucao

Estados e transicoes de uma execucao completa

QUEUED
RUNNING
1/3
STEPS
COMPLETE
FAILED

Queued

executionStateManager.create() cria o registro. Mensagem do usuario salva no banco. Evento emitMessageSaved disparado.

Running

activeExecutions.set() marca como ativo. emitStepStart para cada step. File sync e monitor inicializados.

Steps

Cada step: executeStep() invoca Claude CLI. Stream de eventos (content, actions, metadata) emitido em tempo real via SSE.

Paused

AskUserQuestion detectado. Processo morto, estado salvo com markPaused(). Aguarda resposta do usuario para retomar.

Complete

markCompleted() + evento emitExecutionComplete. Webhook disparado se userId presente. activeExecutions limpo.

Failed

markFailed(errorMessage) + evento emitStepError. Erro pode ser: timeout, step error, exception ou cancel do usuario.
