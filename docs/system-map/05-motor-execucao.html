1 Arquitetura em 7 Camadas

O motor de execucao segue uma arquitetura em 7 camadas bem definidas. Cada camada tem responsabilidade unica e se comunica com a proxima via interfaces tipadas. A separacao garante que mudancas em uma camada nao afetem as demais.

Camada 1: API REST + SSE Streaming execution.routes.ts (706 linhas)

Recebe requisicao do cliente via POST /conversations/:id/messages/stream. Valida autenticacao, carrega workflow e passos, verifica orcamento de tokens via tokenBudgetService.checkBudget(userId).

Headers SSE: Content-Type: text/event-stream, Cache-Control: no-cache, X-Accel-Buffering: no
Timeouts: keepAliveTimeout: 600000ms, requestTimeout: 0 (sem limite para SSE)
Heartbeat: :heartbeat\n\n a cada 15 segundos
↓
Camada 2: BullMQ Queue (Redis) execution-queue.ts + execution-worker.ts

Fila 'execution' com Redis como backend. Workers configuraveis: EXECUTION_CONCURRENCY=3 (env var). Jobs sem retry automatico (attempts: 1). Historico: ultimos 100 completos, 50 falhos.

Dashboard: Bull Board em /admin/queues | Fila secundaria: trigger-scheduler (polling a cada 30s para cron triggers)
↓
Camada 3: Task Orchestrator task-orchestrator.ts (99 linhas)

Facade central que decide qual executor usar. Detecta automaticamente DAG quando steps possuem dependsOn. Gerencia activeExecutions: Map<string, boolean> para evitar execucoes concorrentes na mesma conversa.

Metodos: executeSequential(), executeDAG(), executeStepByStep(), resumeExecution(), cancel(), interrupt()
↓
Camada 4: Workflow Executors 3 executores especializados
Sequential (394 linhas)

Loop while(i < steps.length) com skip, pause, retry, conditions, jump e backoff exponencial

DAG (~180 linhas)

Batches paralelos via Promise.all(), deteccao de ciclos DFS, contexto de dependencias injetado

Step-by-Step (212 linhas)

Um step por chamada, cliente controla avanco via stepIndex, navegacao livre (advance/back/jump)

↓
Camada 5: Step Executor step-executor.ts (410 linhas)

8 fases por step: (1) Recover Session Token, (2) Recover Step Memory, (3) Build System Prompt (base + custom + memory), (4) File Sync, (5) Git Config + GitHub Token, (6) Execute Claude Engine, (7) Handle Result (needsUserInput | interrupted | error | timeout), (8) Save Message & Return.

Recuperacao automatica: session_not_found → delete + retry | prompt_too_long → summarize + compact + retry
↓
Camada 6: Claude Code Engine claude-code-engine.ts (494 linhas)

Spawna processo Claude CLI via sudo -n -u claude-sandbox -g dev --preserve-env com detached: true (process group independente). Parseia stream JSON (NDJSON) em tempo real via StreamParser.

Flags: --print --output-format stream-json --verbose --dangerously-skip-permissions | Opcionais: --resume, --system-prompt, --model
↓
Camada 7: Event System + SSE events.ts (213 linhas)

OrchestratorEvents extends EventEmitter (singleton in-memory). 15 tipos de eventos tipados. SSE handler registra listeners por conversa, filtra por conversationId e serializa como event: name\ndata: json\n\n.

Multi-client: cada conexao SSE tem listener set independente | Cleanup automatico no disconnect: remove listeners + cancela execucao

2 Fluxo de Execucao Completo

Desde o momento em que uma mensagem chega ate a conclusao da execucao, cada etapa e cuidadosamente orquestrada. Fluxo detalhado com 11 etapas:

POST /conversations/:id/messages/stream

Requisicao com { content, stepIndex?, attachments? }. Body validado via Zod schema. O stepIndex e opcional (usado para step-by-step).

Validacao e Carregamento

Carrega conversa com workflow + steps (ordenados por stepOrder ASC) via Prisma. Valida existencia, permissoes e tipo de workflow. Se step_by_step, resolve currentStepId ou usa stepIndex explicitamente.

Verificacao de Orcamento

tokenBudgetService.checkBudget(userId) — verifica saldo de tokens. Retorna { allowed: boolean, reason?: string }. Se bloqueado, HTTP 429 imediato.

Cancelamento de Execucao Stale

Se taskOrchestrator.isExecuting(id), forca cancelamento: SIGTERM → pkill filhos → SIGKILL apos 2s. Aguarda 500ms para cleanup. Recovery no startup: todas execucoes state: 'running' orfas sao marcadas como failed.

Setup SSE + Listener Registration

SSE headers configurados. Heartbeat a cada 15s. 15 event listeners registrados no orchestratorEvents: step:start, step:stream, step:complete, step:error, message:saved, condition:retry, condition:jump, execution:complete, context:reset, execution:cancelled, dag:batch_start, validation:failed, execution:paused, execution:resumed, user:interrupt. Cada listener filtra por conversationId.

Deteccao de Execucao Pausada

taskOrchestrator.getPausedExecution(id) — se ha execucao pausada (AskUserQuestion), redireciona para resumeExecution() com a resposta do usuario ao inves de iniciar nova execucao.

Project Path Lock + Enqueue

Adquire lock no projectPath (evita conflitos de IO). Job enfileirado com: conversationId, workflowId, userId, content, stepIndex, projectPath, workflowType, attachments.

Worker Processa Job

processExecutionJob carrega conversa do DB, monta ExecutionContext, e decide executor: isDAGWorkflow(steps) → DAG | workflowType === 'sequential' → Sequential | else → StepByStep.

Execucao dos Steps

Cada step: (1) skip check, (2) emit step:start, (3) executeStep(), (4) stream events em tempo real, (5) handle result (pause/error/success), (6) save message, (7) evaluate conditions, (8) advance/retry/jump/finish. Validators opcionais com retry automatico.

Checkpoint & Save

Apos cada step: saveCheckpoint(executionId, stepIndex, output, nextInput), message persistida no DB com role: 'assistant', metadata incluindo actions, sessionId, stepName, stepOrder.

Finalizacao

markCompleted(executionId) → emitExecutionComplete({success: true}) → SSE envia event: complete → cleanup de listeners → activeExecutions.delete(conversationId). Dispara webhook execution:complete se userId configurado.

sequential-executor.ts

394 linhas | Executa passos em ordem com full control flow

// ID: exec_{conversationId}_{timestamp}_{random5}
const executionState = await executionStateManager.create(conversationId, startIndex)
let i = startIndex, currentInput = userInput
const retryCounts: Record<string, number> = {}

while (i < steps.length) {
  // 1. Cancellation check
  if (!deps.activeExecutions.get(conversationId)) {
    emitExecutionCancelled(); markCancelled(); return
  }

  // 2. Skip condition (6 formatos suportados)
  if (step.skipCondition) {
    if (evaluateSkipCondition(step.skipCondition, currentInput, i)) {
      emitStepStream({ 'Step pulado' }); i++; continue
    }
  }

  // 3. Execute step (8 fases internas)
  emitStepStart({ stepId, stepName, stepOrder: i+1, totalSteps })
  let result = await executeStep(deps, executionId, ...)

  // 4. AskUserQuestion → pause
  if (result.needsUserInput) {
    const q = extractAskQuestion(result.actions)
    saveAssistantMessage({ needsUserInput: true })
    markPaused(executionId, i, step.id, result.resumeToken, q)
    emitExecutionPaused({ askUserQuestion: q })
    return // exit - aguarda resposta do usuario
  }

  // 5. Error handling (4 estrategias)
  if (isStepError(result)) {
    const decision = decideOnError(step, result)
    if (decision.action === 'fail') { markFailed(); return }
    if (decision.action === 'fallback') { retryWithFallbackMessage() }
    if (decision.skipStep) { i++; continue }
  }

  // 6. Validators (optional, maxRetries default 2)
  if (step.validators) { validateAndRetry() }

  // 7. Save assistant message + checkpoint
  saveMessage(conversationId, step.id, result.content)
  saveCheckpoint(executionId, i, result.content)

  // 8. Conditions evaluation (8 tipos, 5+ acoes)
  const condResult = conditionsEvaluator.evaluate(result.content, step.conditions)
  const nextStep = resolveNextStep(condResult.action, steps, i)
  if (nextStep.isRetry) { handleRetryWithBackoff() }
  if (nextStep.isFinished) { emitComplete(); break }
  i = nextStep.nextIndex; currentInput = result.content
}

markCompleted(executionId)
emitExecutionComplete({ success: true })

Skip Conditions (6 formatos)

"true"

Sempre pula o step

"false"

Nunca pula

"empty_input"

Pula se input vazio

"contains:keyword"

Pula se input contem keyword (case-insensitive)

"not_contains:keyword"

Pula se input NAO contem keyword

"step_index > 3"

Operadores: >, <, >=, <=, ==

Exponential Backoff

Formula: Math.min(Math.pow(2, attempt) * 1000, 30000)

Attempt 1: 2s Attempt 2: 4s Attempt 3: 8s Attempt 4: 16s Attempt 5+: 30s (cap)

Retry counts armazenados em ExecutionState.retryCounts: Record<string, number>. Keys: step.id (condition retry), validator_{step.id} (validators), jump_{stepId}_to_{nextIndex} (jumps). Default maxRetries: 3 (condition) | 2 (validator).

4 DAG Executor (Parallel)

◇

Directed Acyclic Graph

dag-executor.ts + dag-workflow-executor.ts | Execucao paralela com dependencias

graph: Map<string, DagStep>

Adjacency list. Cada node: {step, index, dependsOn: string[]}

completed: Set<string>

O(1) lookup para steps ja completados

outputs: Map<string, string>

Armazena output de cada step para injecao de contexto

// 1. Build DAG (O(V) onde V = steps)
for (const step of steps) {
  graph.set(step.id, { step, index, dependsOn: parseDependsOn(step.dependsOn) })
}

// 2. Validate - DFS cycle detection (O(V+E))
const visited = new Set(), inStack = new Set()
function hasCycle(id) {
  if (inStack.has(id)) return true  // Back edge = cycle!
  if (visited.has(id)) return false  // Already explored
  visited.add(id); inStack.add(id)
  for (const dep of node.dependsOn) { if (hasCycle(dep)) return true }
  inStack.delete(id); return false
}

// 3. Execute in batches (implicit topological sort)
let batchIndex = 0
while (!dag.isComplete()) {
  const readySteps = dag.getReadySteps() // deps satisfied && not completed
  if (readySteps.length === 0) break // Deadlock detection
  emitDagBatchStart({ stepIds, batchIndex })

  // 4. Parallel execution (unlimited concurrency per batch)
  const results = await Promise.all(
    readySteps.map(async (dagStep) => {
      const depContext = dag.getDependencyContext(dagStep.step.id)
      const input = depContext ? `${userInput}\n\n${depContext}` : userInput
      return { dagStep, result: await executeStep(..., input) }
    })
  )

  // 5. Process results + mark completed
  for (const { dagStep, result } of results) {
    if (isStepError(result) && resolveErrorHandler(dagStep.step) === 'fail') {
      markFailed(); return // Stop entire DAG
    }
    dag.markCompleted(dagStep.step.id, result.content)
  }
  batchIndex++
}
Injecao de Contexto de Dependencias

getDependencyContext(stepId) concatena outputs dos steps-pai:

// Formato do contexto injetado:
[Output from "Step A"]:
...output completo do step A...

[Output from "Step B"]:
...output completo do step B...
A (sem deps)
→
B (deps: A)
→
D (deps: B, C)
A
→
C (deps: A)
→
D

B e C executam em paralelo (Batch 2). D so executa apos ambos completarem (Batch 3).

Error Handling no DAG

Erros sao avaliados per-step durante o batch. Se errorHandler === 'fail', a execucao inteira para apos o batch corrente completar (Promise.all aguarda todos). Para skip/continue_next, o step e marcado como completed e o DAG continua. Para fallback, re-executa com FALLBACK_MESSAGE antes de retornar resultado.

private buffer = ''

parse(chunk: string): StreamEvent[] {
  this.buffer += chunk         // Accumulate
  const lines = this.buffer.split('\n')
  this.buffer = lines.pop() || '' // Keep incomplete last line

  for (const line of lines) {
    try {
      const data = JSON.parse(line.trim())
      events.push(...this.parseJsonEvent(data))
    } catch {
      events.push({ type: 'content', content: line.trim() }) // Fallback
    }
  }
}

10 Pausa e Resumption

Quando Claude chama AskUserQuestion, o motor detecta via USER_INPUT_TOOLS = new Set(['AskUserQuestion']), mata o processo, salva estado e aguarda input do usuario. Lifecycle completo:

1 Claude chama AskUserQuestion (stream)

StreamParser emite action com type: 'tool_use', name: 'AskUserQuestion'. Engine detecta via USER_INPUT_TOOLS.has(action.name). Flag needsUserInput = true.

↓
2 SIGTERM imediato ao process group

process.kill(-pid, 'SIGTERM') mata grupo inteiro. Fallback: childProcess.kill('SIGTERM'). Apos 2000ms: SIGKILL forcado.

↓
3 Extracao da pergunta + Estado salvo

Itera result.actions buscando AskUserQuestion. Extrai questions[0].question e questions[0].options[]. Chama markPaused(executionId, stepIndex, stepId, resumeToken, askUserQuestion). Metadata: pausedAt, pausedStepIndex, pausedStepId, pausedResumeToken, askUserQuestion.

↓
4 SSE: execution_paused + Aguarda resposta

emitExecutionPaused({ askUserQuestion: { question, options } }). Cliente recebe pergunta e opcoes. activeExecutions.delete(conversationId). Status endpoint: GET /conversations/:id/status → { isPaused: true, pausedInfo }.

↓
5 Usuario responde → resumeExecution()

POST /conversations/:id/messages/stream com resposta. Route detecta getPausedExecution(id) e redireciona para resumeExecution(deps, context, userAnswer, pausedInfo). Re-executa MESMO step com resposta como input + --resume token. Se Sequential: auto-continua restante dos steps. Se Step-by-Step: para apos step.

Cenarios de Erro Especiais

Timeout

withTimeout(promise, stepTimeoutMs). Default: 0 (sem timeout). Timeout → engine.cancel() → result com timedOut: true. Formula: Step "X" excedeu o timeout de Ys.

Session Not Found

Sessao Claude expirou. isSessionNotFoundError() detecta. Recovery: deleteSession() → retry sem --resume + emit context:reset.

Prompt Too Long

isPromptTooLongError() detecta (case-insensitive). Recovery: summarize via Haiku → delete session → retry com memory compactada.

Process Crash (exit code != 0)

Stderr heuristics: Error:, ENOENT, EACCES, Cannot find, Authentication, API key. Error enrichment: se erro generico, appenda stderr (ate 1000 chars).

starts_with — prefixo (case-insensitive)
ends_with — sufixo (case-insensitive)
regex — RegExp com flag 'i'. Erros de regex sao logados e retornam false
length_gt — content.length > parseInt(match)
length_lt — content.length < parseInt(match)

5+ Acoes Disponiveis

next — proximo step (i+1). isFinished se ultimo.
previous — step anterior. Math.max(0, i-1).
retry — re-executa mesmo step (isRetry: true). Backoff exponencial.
finish — encerra workflow (nextIndex = steps.length).
{stepId} — jump por ID. Busca no array de steps.
{number} — jump por indice numerico. Fallback se ID nao encontrado.

{ type, match, goto,
  maxRetries?: number, // Override por regra
  retryMessage?: string // Template: {output}, {error}, {type}
}

16 Cancellation & Interrupt

Preserva sessoes. Salva output parcial + mensagem do usuario. Pode continuar.
