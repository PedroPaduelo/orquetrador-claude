1 Arquitetura em Camadas

O motor de execucao segue uma arquitetura em camadas, onde cada camada tem uma responsabilidade especifica e bem definida. A API REST recebe as requisicoes, a fila BullMQ gerencia a concorrencia, o Task Orchestrator coordena a execucao, e finalmente o Claude Code Engine executa os comandos.

API REST + SSE Streaming POST /conversations/:id/messages/stream

Recebe a requisicao do cliente, valida autenticacao, carrega workflow e passos, verifica orcamento de tokens e estabelece conexao Server-Sent Events para streaming em tempo real.

BullMQ Queue (Redis) concurrency=3

Fila de execucao com Redis como backend. Configurada com 3 workers simultaneos. Job name: "execution". Gerencia prioridade, retries e estado de processamento.

Task Orchestrator Router

Decide qual executor usar baseado no tipo de workflow: Sequential, DAG ou Step-by-Step. Gerencia estados de execucao e emite eventos para SSE.

[Sequential | DAG | Step-by-Step] Executors

Sequential: Executa passos em ordem, com suporte a pause, retry e jump.
DAG: Executa passos em paralelo baseado em dependencias.
Step-by-Step: Executa um passo por vez, controlado pelo cliente.

Step Executor Step Engine

Prepara o contexto de execucao: recupera sessao, memoria, constroi system prompt, sincroniza arquivos, configura git e invoca o Claude Code Engine.

Claude Code Engine claude CLI

Spawna processo Claude CLI via sudo -u claude-sandbox, parseia stream JSON, gerencia AskUserQuestion, timeouts e cancelamentos.

Redis PubSub + SSE Event Streaming

Eventos sao publicados no Redis PubSub e transmitidos via SSE para o cliente. Heartbeat a cada 15s mantem a conexao viva.

Sequential Executor

Executa passos em ordem sequencial, um apos o outro

while (i < steps.length) {
  // 1. Skip check
  if (skipCondition(step)) { i++; continue; }
  if (evaluateSkip(step, context)) { i++; continue; }

  // 2. Execute step
  const result = await executeStep(step, userInput);

  // 3. Handle pause (AskUserQuestion)
  if (result.needsUserInput) {
    markPaused(executionId, i, stepId, resumeToken, question);
    return;
  }

  // 4. Handle error
  const action = decideOnError(error, step.decideOnError);
  if (action === 'fail') markFailed();
  if (action === 'skip') { i++; continue; }
  if (action === 'continue_next') { i++; continue; }
  if (action === 'fallback') retryWithFallback();

  // 5. Save message
  await saveMessage(conversationId, stepId, result.content);

  // 6. Evaluate conditions
  const condition = evaluateConditions(result, step.conditions);
  if (condition.action === 'jump') i = condition.targetIndex;
  if (condition.action === 'retry') continue;
  if (condition.action === 'finish') break;

  i++; // next step
}

// 7. Finalize
markCompleted(executionId);
emitExecutionComplete(executionId, stepsCompleted);

Caracteristicas

  • • Execucao deterministica em ordem
  • • Suporte a skip condicional
  • • Pausa em AskUserQuestion
  • • 4 estrategias de tratamento de erro
  • • Avaliacao de condicoes pos-step
  • • Jump para step arbitrario
  • • Retry com contador maximo

DAG Executor

Directed Acyclic Graph - Execucao paralela baseada em dependencias

// 1. Build DAG from steps
const dag = new DAG(steps, {
  getId: (step) => step.id,
  getDependencies: (step) => step.dependsOn || []
});

// 2. Validate cycles (DFS)
if (dag.hasCycle()) {
  throw new Error('Circular dependency detected');
}

// 3. Execute until complete
while (!dag.isComplete()) {
  // 4. Get ready steps (deps satisfied)
  const readySteps = dag.getReadySteps();

  // 5. Parallel execution
  const results = await Promise.all(
    readySteps.map(step =>
      executeStep(step, getDependencyContext(step))
    )
  );

  // 6. Mark completed
  results.forEach(r => dag.markCompleted(r.stepId, r.output));
}

// 7. Finalize
markCompleted(executionId);
emitExecutionComplete(executionId, stepsCompleted);

Caracteristicas

  • • Execucao paralela de steps independentes
  • • Deteccao de ciclos via DFS
  • • Contexto de dependencias injetado
  • • Ordem topologica garantida
  • • Erro em um step nao bloqueia outros

Step-by-Step Executor

Execucao controlada pelo cliente - um step por vez

// Execute single step by index
async function executeStepByStep(context, userInput, stepIndex) {
  const step = context.workflow.steps[stepIndex];

  // Same error handling as sequential
  const result = await executeStep(step, userInput);

  // Handle pause via AskUserQuestion
  if (result.needsUserInput) {
    markPaused(executionId, stepIndex, step.id, resumeToken, question);
    SIGTERM process; // kill cleanly
    return;
  }

  // Save and emit
  saveMessage(conversationId, step.id, result.content);
  emitStepComplete(step.id, result);
}

Caracteristicas

  • • Cliente controla o avanco
  • • Um step executado por chamada
  • • Pausa automatica em AskUserQuestion
  • • Mesmo tratamento de erro que sequential
  • • Estado preservado entre chamadas
  • • Resume via --resume token

4 Step Executor Deep Dive

O Step Executor e o coracao de cada execucao. Ele prepara o contexto, invoca o Claude Code Engine e processa o resultado. Cada passo passa por estas fases:

1. Recover Session Token

sessionManager.getSession(conversationId, stepId)

Recupera o resumeToken da sessao anterior para continuidade.

2. Recover Step Memory

memoryManager.getMemory(conversationId, stepId)

Recupera o resumo do contexto anterior para compactacao.

3. Build System Prompt

buildSystemPrompt(base + custom + memory)

Combina prompt base, prompt customizado do step e memoria injetada.

4. File Sync

fileSyncService.sync(projectPath)

Sincroniza arquivos do projeto com o ambiente de execucao.

5. Git Config

fetch GitHub token + git config

Configura credenciais Git para operacoes versionadas.

6. Execute Claude Engine

claudeCodeEngine.execute({...})

Invoca o Claude CLI com timeout configuravel.

7. Handle Result

needsUserInput | interrupted | error | timeout

Processa cada tipo de resultado apropriadamente.

8. Save & Return

saveMessage() + return result

Persiste a mensagem e retorna o conteudo gerado.

9 Tratamento de Erros

Cada step pode configurar como erros devem ser tratados via decideOnError. Esta estrategia define o comportamento quando uma excecao ocorre.

fail Padrao

Para a execucao imediatamente e marca como FAILED. Erro e propagado para o cliente via SSE.

skip

Ignora o erro e continua para o proximo step. Nenhuma mensagem de erro e enviada ao cliente.

continue_next

Trata o output de erro como sucesso e prossegue. Mensagem de erro e incluida no conteudo.

fallback

Retenta com mensagem simplificada: "Retry: a execucao anterior falhou..."

Cenarios de Erro Especiais

⏱ Timeout

Configuravel por step via timeout (default: 10min). Processo recebe SIGTERM → aguarda 2s → SIGKILL.

🔍 Session Not Found

Sessao Claude expirou ou foi invalidada. Delete session → re-executa sem --resume.

📏 Prompt Too Long

Contexto excede limite de tokens do modelo. Compact memory → reset session → re-execute.

🛑 Cancellation

Usuario ou sistema cancela a execucao. SIGTERM para process group → cleanup de recursos.

10 Conditions Evaluator

Regras condicionais podem ser definidas por step para controlar o fluxo de execucao baseado no conteudo gerado. Isso permite workflows dinamicos e adaptativos.

// Condition structure
interface Condition {
  type: 'contains' | 'not_contains' | 'equals' | 'regex';
  value: string;
  action: 'next' | 'retry' | 'jump-to-'<stepId> | 'finish' | 'previous';
  maxRetries?: number;
  retryMessage?: string;
}

Types

  • contains - valor presente no conteudo
  • not_contains - valor ausente
  • equals - conteudo exatamente igual
  • regex - expressao regular casa

Actions

  • next - proximo step
  • retry - repete mesmo step
  • jump-to-- proximo step especifico
  • finish - encerra workflow
  • previous - volta ao anterior

Exemplo de Uso

conditions: [
  { type: 'contains', value: 'ERROR', action: 'jump-to-error-handler' },
  { type: 'regex', value: '^DONE:', action: 'finish' },
  { type: 'contains', value: 'retry', action: 'retry', maxRetries: 3 }
]

14 Melhorias Recomendadas (2026)

Pesquisa de arquitetura identificou oportunidades de melhoria no motor de execucao. Abaixo, as recomendacoes priorizadas por impacto.

Prompt Caching

ALTA

Anthropic prompt cache com tokens de break point (__bp__). Reducao estimada de 90% em custo de input tokens e 80% em latencia.

Impact: -90% custo | -80% latencia

Circuit Breaker

ALTA

Protecao contra falhas sistemicas (API down, rate limit). Estados: Closed → Open → Half-Open com thresholds configuraveis.

Impact: Resiliencia | Prevencao de cascata

Structured Outputs

ALTA

JSON Schema validation por step. Previne drift entre steps e garante que outputs atendam contratos definidos.

Impact: Consistencia | Type safety

Filesystem Polling

MEDIA

Output em arquivo ao inves de inline. Reduz complexidade de O(n*output) para O(n*filename). Polling detecta conclusao.

Impact: -50% contexto | Maior escalabilidade

Concorrencia Dinamica DAG

MEDIA

maxConcurrency configuravel por DAG. Fila com prioridade. Critical path first para otimizar tempo total.

Impact: -40% tempo DAG | Melhor throughput

Metricas Agregadas

MEDIA

Taxa sucesso/erro por workflow, latencia P50/P95, custo medio, bottleneck analysis. Dashboards para monitoramento.

Impact: Observabilidade | Optimizacoes

Verificacao Semantica

BAIXA

LLM rapido (Haiku) valida qualidade do output antes de prosseguir. Checa se requisitos foram atendidos, se ha contradicoes.

Impact: Qualidade | +20% custo

Estado Estruturado

BAIXA

Schemas tipados para inputs/outputs entre steps. Contratos explicitos com validacao Zod. Previne propagacao de dados malformados.

Impact: Type safety | Debugging
