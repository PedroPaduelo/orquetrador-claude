⏰ O que sao Triggers

Triggers sao gatilhos automatizados que iniciam a execucao de workflows sem intervencao manual. Cada trigger esta vinculado a um workflow e, quando ativado, cria uma ScheduledExecution, que por sua vez gera uma Conversation e enfileira o job no BullMQ para execucao pelo TaskOrchestrator.

O sistema suporta 5 tipos de trigger, cada um com seu mecanismo de ativacao: manual (usuario), cron (agendamento), webhook (chamada HTTP externa), event (evento nomeado) e git_push (push no GitHub). Todos respeitam rate limiting configuravel (triggers/hora) e podem ser habilitados/desabilitados individualmente.

⚡ 5 Tipos de Trigger

👆
manual
user-initiated

Disparado pelo usuario via UI ou API. O endpoint POST /triggers/:id/fire executa imediatamente.

Campos: nenhum campo adicional obrigatorio
🕐
cron
scheduled

Execucao agendada via expressao cron. Suporta timezone para agendamentos localizados. Reagenda automaticamente apos cada execucao.

Campos: cronExpr (obrigatorio), cronTimezone
🪝
webhook_inbound
external

Chamada HTTP externa dispara o workflow. Um webhookSecret (32 bytes hex) e gerado automaticamente na criacao para autenticacao.

Campos: webhookSecret (auto-gerado, criptografado com AES)
📡
event
named-event

Evento nomeado interno dispara o workflow. Suporta filtro JSON para matchear somente eventos com payload especifico.

Campos: eventName (obrigatorio), eventFilter (JSON)
📂
git_push
github

Push no repositorio GitHub dispara o workflow automaticamente. Ideal para CI/CD pipelines e deploys automatizados.

Campos: usa webhook do GitHub para detectar eventos push

⚙️ Trigger Scheduler (Backend)

🔄
trigger-scheduler.ts

Worker BullMQ que roda como repeatable job a cada 30 segundos. Busca ScheduledExecutions pendentes (status: pending, scheduledAt <= now), verifica rate limit, cria conversation e delega execucao ao TaskOrchestrator.

Ciclo de Execucao

  1. Busca execucoes due via findDueScheduled()
  2. Marca como running
  3. Atualiza lastTriggeredAt no trigger
  4. Verifica rate limit (count execucoes na ultima hora)
  5. Busca workflow completo com steps
  6. Cria Conversation com titulo [Trigger] nome - data
  7. Executa via TaskOrchestrator (DAG, Sequential ou StepByStep)
  8. Marca como completed ou failed
  9. Se cron: re-agenda proxima execucao via scheduleCron()

Configuracao

Queue name: trigger-scheduler
Poll interval: 30.000ms (30 segundos)
Concurrency: 1 (serial)
removeOnComplete: ultimos 50 jobs
removeOnFail: ultimos 20 jobs
Cron parser: cron-parser lib
Startup: startTriggerScheduler() no server.ts
Shutdown: stopTriggerScheduler() graceful
🛡️
Rate Limiting

Cada trigger pode ter um rateLimit (triggers/hora). O scheduler conta execucoes com status completed ou running na ultima hora. Se o limite for atingido, a execucao e marcada como failed com mensagem Rate limit exceeded (N/hour).

async function countRecentTriggers(triggerId, rateLimit) { const oneHourAgo = new Date(Date.now() - 60 * 60 * 1000) return prisma.scheduledExecution.count({ where: { triggerId, executedAt: { gte: oneHourAgo }, status: { in: ['completed', 'running'] }, }, }) }

🔌 API Endpoints (7)

GET /workflows/:workflowId/triggers List triggers (paginated)
▼
Auth: JWT (Bearer token)
Params: workflowId (path)
Query: page, limit (pagination)
Response: { data: Trigger[], meta: { page, limit, total, totalPages } }
Bonus: Inclui scheduledCount em cada trigger
// Exemplo curl curl -s "https://orquestrador-api.../workflows/WORKFLOW_ID/triggers?page=1&limit=10" \ -H "Authorization: Bearer $TOKEN" | jq .
GET /triggers/:id Get trigger detail
▼
Auth: JWT (Bearer token)
Params: id (trigger ID)
Response: Trigger object completo (webhookSecret descriptografado)
Ownership: Verifica workflow.userId
POST /workflows/:workflowId/triggers Create trigger
▼
Auth: JWT (Bearer token)
Response: 201 Created
Validacoes:
  • cron → cronExpr obrigatorio
  • event → eventName obrigatorio
  • webhook_inbound → webhookSecret gerado automaticamente (32 bytes hex, AES encrypted)
// Body { "type": "cron", "cronExpr": "0 9 * * 1-5", "cronTimezone": "America/Sao_Paulo", "rateLimit": 10, "enabled": true }
PUT /triggers/:id Update trigger
▼
Auth: JWT (Bearer token)
Ownership: Verifica em $transaction antes de atualizar
Campos atualizaveis: cronExpr, cronTimezone, eventName, eventFilter, rateLimit, enabled
// Body parcial { "cronExpr": "*/10 * * * *", "enabled": false }
DELETE /triggers/:id Delete trigger
▼
Auth: JWT (Bearer token)
Response: 204 No Content
Ownership: Verifica em $transaction (findFirst + delete atomico)
GET /triggers/:id/scheduled List scheduled executions
▼
Auth: JWT (Bearer token)
Query: page, limit
Response: Paginated ScheduledExecution list (orderBy scheduledAt desc)
POST /triggers/:id/fire Manually fire trigger
▼
Auth: JWT (Bearer token)
Response: 201 (ScheduledExecution criada)
Validacao: Trigger deve estar enabled: true
Efeito: Cria ScheduledExecution com scheduledAt: now via triggersService.fireNow()
// Dispara manualmente curl -s -X POST "https://orquestrador-api.../triggers/TRIGGER_ID/fire" \ -H "Authorization: Bearer $TOKEN" | jq .

🔒 Seguranca

🔑
Webhook Secret

Gerado com crypto.randomBytes(32).toString('hex'). Armazenado criptografado com AES via encrypt(). Descriptografado com decrypt() ao ler.

👤
Ownership

Todas as operacoes verificam workflow.userId. Update e Delete usam $transaction para atomicidade (findFirst + operacao no mesmo escopo).

🛡️
Rate Limit

Configuravel por trigger (rateLimit = max triggers/hora). Contagem de execucoes completed/running na ultima hora. Excedeu? Status failed.
