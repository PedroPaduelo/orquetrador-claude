1 Visao Geral

Workflows sao templates de execucao do Orquetrador. Cada workflow define uma sequencia de steps que serao executados em ordem, com suporte a conditions, retries, validadores e integracao com recursos (MCP servers, skills, agents, rules, hooks).

sequential
Todos os steps executam em sequencia, um apos o outro. Se o workflow tiver dependsOn em algum step, o motor de execucao usa DAG automaticamente.
step_by_step
Executa um step por vez, aguardando input do usuario entre cada step. Ideal para fluxos interativos e revisao humana.
Conceitos-chave
  • Workflow — template de execucao com metadata e steps
  • WorkflowStep — unidade atomica de execucao dentro do workflow
  • Conditions — regras que determinam o proximo step com base no output
  • Conversation — instancia de execucao de um workflow
  • WorkflowVersion — snapshot + diff para rollback
  • StepTemplate — configuracao reutilizavel de step

4 WorkflowStep — Anatomia Completa

Campos Base ▼
nameString
Nome identificador do step.
baseUrlString
URL base do projeto para execucao (path no servidor).
stepOrderInt
Posicao do step na sequencia (0-indexed).
systemPromptString?
Prompt de sistema customizado para este step.
useBasePromptBoolean
Se true, combina com o prompt base do usuario.
Default: true
Conditions — Sistema de Regras ▼

O campo conditions e um JSON com regras que avaliam o output do step e determinam a acao seguinte.

{ "rules": [ { "type": "contains", // tipo de comparacao "match": "APROVADO", // valor para comparar "goto": "complete", // acao: next|retry|jump|complete "maxRetries": 3, // tentativas (se retry) "retryMessage": "Tente novamente com mais detalhe" } ], "default": "next" // acao se nenhuma regra matched }
maxRetriesInt
Numero maximo de retries para o step.
Default: 0
DAG Dependencies ▼
dependsOnJson (string[])
Array de IDs de steps dos quais este step depende. Se preenchido, o motor usa DAG Executor para execucao paralela.
Default: []

Quando qualquer step tem dependsOn preenchido, o workflow e automaticamente detectado como DAG pelo isDAGWorkflow() e executado com o DAG Executor.

Validators ▼
validatorsJson (array)
Array de validadores que checam o output do step antes de prosseguir. Cada validador tem type e config.
Default: []
"validators": [ { "type": "json_schema", "config": { "schema": { ... } } }, { "type": "contains", "config": { "text": "OK" } }, { "type": "min_length", "config": { "length": 100 } } ]
Timeout e Error Handling ▼
timeoutInt? (ms)
Tempo maximo de execucao do step em milissegundos.
Default: 300000 (5 min)
errorHandlerStepErrorHandler
Estrategia quando o step falha.
Default: fail
skipConditionString?
Expressao avaliada em runtime. Se true, o step e pulado.
Enum StepErrorHandler
fail
Para a execucao inteira
skip
Pula o step e continua
fallback
Executa logica de fallback
continue_next
Ignora erro e prossegue
Resources (Recursos Vinculados) ▼

Cada step pode ter multiplos recursos vinculados via join tables (M:N).

🤖 mcpServerIds WorkflowStepMcpServer
🧠 skillIds WorkflowStepSkill
🤖 agentIds WorkflowStepAgent
📐 ruleIds WorkflowStepRule
🪝 hookIds WorkflowStepHook

Cada join table tem: id, stepId, resourceId, order, config?, createdAt. Unique constraint em [stepId, resourceId].

Variables e Output ▼
outputVariablesJson (string[])

Default: []
inputVariablesJson (string[])
Variaveis esperadas como input (de steps anteriores).
Default: []
outputSchemaJson?

Backend e Model ▼
backendString

Modelo especifico para o step (ex: claude-3-opus, claude-3-sonnet). Null usa o padrao.

Fluxo de Versionamento
v1 — Criacao
Snapshot inicial ao criar o workflow. Sem diff.
v2 — Edicao de step
Novo snapshot criado automaticamente ao salvar (PUT /workflows/:id). Diff e changelog calculados.
v3 — Rollback para v1
POST /workflows/:id/versions/rollback {version: 1}. Restaura o workflow a partir do snapshot da v1. Cria uma nova versao (v3) com changelog indicando rollback.

8 Step Templates

Modelo Prisma: StepTemplate @@map("step_templates")

Templates reutilizaveis de configuracao de step. Permitem criar steps padronizados sem reconfigurar tudo do zero. Unique constraint em [name, userId].

nameString
Nome do template (unico por usuario).
descriptionString?
Descricao do proposito do template.
baseUrlString
URL base padrao do template.
Default: ""
systemPromptString?
Prompt de sistema padrao do template.
conditionsJson
Conditions padrao do template.
Default: {}
resourceIdsJson
Mapa de IDs de recursos pre-configurados.
Default: {}
Quando um step referencia um template via templateId, ele herda as configuracoes do template. O step pode sobrescrever qualquer campo herdado.

10 Workflow Wizard (Frontend)

O wizard de criacao de workflows no frontend e divido em 3 fases. O usuario pode navegar entre as fases livremente e a criacao so e efetivada ao finalizar.

1
Basic Info
  • Nome do workflow
  • Descricao (opcional)
  • Tipo: sequential ou step_by_step
  • Tags para categorizacao
2
Step Builder
  • Adicionar/remover steps
  • Editar nome, prompt, baseUrl por step
  • Reordenar steps (drag & drop)
  • Vincular recursos (MCP, Skills, Agents, Rules, Hooks)
  • Configurar conditions e validators
  • Definir error handler e timeout
  • Selecionar template como base
3
Review
  • Preview visual do fluxo completo
  • Resumo de steps e conditions
  • Visualizacao de resources vinculados
  • Botao de criar/salvar workflow
