1 Visao Geral

Arquitetura e conceitos fundamentais do sistema de workflows

Workflows sao templates de execucao do Orquetrador Claude. Cada workflow define uma sequencia de steps que serao executados pelo motor de execucao, com suporte a conditions (roteamento condicional), retries (re-execucao automatica), validators (validacao de output), DAG paralelo (execucao com dependencias) e integracao profunda com todos os recursos do sistema (MCP servers, skills, agents, rules, hooks).

▶ SEQUENTIAL

Steps executam em ordem linear (0 → 1 → 2...). Conditions determinam o proximo step. Se algum step tiver dependsOn, o motor detecta automaticamente e redireciona para DAG.

type: "sequential" dependsOn: []
⚖ DAG PARALELO

Grafo Aciclico Dirigido. Steps independentes executam em paralelo. Steps com dependsOn aguardam dependencias. O DAGExecutor detecta ciclos, valida referencias e orquestra batches.

type: "sequential" dependsOn: [...]
✋ STEP-BY-STEP

Executa UM step por vez, aguardando input do usuario entre cada step. Ideal para fluxos interativos, revisao humana e processos que requerem aprovacao entre etapas.

type: "step_by_step"

Conditions — Sistema de Regras 8 tipos | 4 acoes
▼

O campo conditions e um JSON com array de rules + acao default. O ConditionsEvaluator avalia cada regra em ordem contra o output do step. Todas as comparacoes (exceto regex e length) normalizam para lowercase. A regex usa flag i (case-insensitive) com try/catch para expressoes invalidas.

{ "rules": [ { "type": "contains", // tipo de comparacao (8 tipos) "match": "APROVADO", // valor para comparar com o output "goto": "complete", // acao: next | retry | finish | step-id | step-index "maxRetries": 3, // tentativas maximas (se retry) "retryMessage": "Tente novamente com mais detalhe. Output: {output}" } ], "default": "next" // acao se nenhuma rule matched }
maxRetriesInt
Numero maximo de retries para este step. Usado quando condition ou validator triggam retry. Cada retry usa backoffDelay() para espera exponencial.
Default: 0 (sem retries) | Max recomendado: 5
DAG Dependencies Execucao paralela
▼
dependsOnJson (string[])
Array de IDs de steps dos quais este step depende. Quando qualquer step tem dependsOn preenchido, o workflow e detectado como DAG pelo isDAGWorkflow() e o TaskOrchestrator redireciona para o DAGWorkflowExecutor.
Default: [] (vazio = sem dependencias = sequencial)
Auto-deteccao: A funcao isDAGWorkflow(steps) percorre todos os steps. Se qualquer um tiver dependsOn.length > 0, o workflow inteiro e executado em modo DAG, mesmo sendo type="sequential". Steps sem dependsOn sao tratados como nodos raiz e executam imediatamente no primeiro batch.

▼

Default: []

{ "type": "json_schema", "schema": { ... } }

6 Modos de skipCondition

"true" — Sempre pula
"false" — Nunca pula
"empty_input" — Pula se input vazio/trim
"contains:KEYWORD" — Pula se input contem (lowercase)
"not_contains:KEYWORD" — Pula se NAO contem
"step_index > N" — Pula por comparacao de indice (>, <, >=, <=, ==)

Enum StepErrorHandler

fail
Para a execucao inteira. Marca ExecutionState como failed. Emite execution:failed. Padrao seguro.
skip
Pula o step sem salvar output. Continua para o proximo step. Emite step:skip no log.
fallback
Re-executa com FALLBACK_MESSAGE simplificado: "Retry: a execucao anterior falhou. Simplifique a abordagem."
continue_next
Trata erro como sucesso. Continua normalmente. Output de erro e passado para o proximo step.

5 Motor de Execucao — 3 Modos

O TaskOrchestrator analisa o workflow e roteia para o executor correto

▶

SequentialExecutor

sequential-executor.ts

Loop principal que percorre steps em ordem. Para cada step: verifica skipCondition, avalia conditions do step anterior, executa step, roda validators, avalia conditions do output, determina proximo step.

✓ Loop while(!complete) com stepIndex
✓ Conditions evaluation apos cada step
✓ Retry com backoff exponencial
✓ Validator check + retry on fail
✓ Error handler per-step (fail/skip/fallback/continue)
✓ Pause/resume via AskUserQuestion
✓ Checkpoint save per step para recovery
// Pseudocodigo SequentialExecutor while (!complete && stepIndex < steps.length) { step = steps[stepIndex] // 1. Skip condition if (step.skipCondition && eval(skipCondition)) stepIndex++; continue // 2. Execute step result = await executeStep(step, input) // 3. Error handling if (isStepError(result)) { decision = decideOnError(step, result) // fail | skip | fallback | continue_next } // 4. Run validators await runValidators(step.validators, result) // 5. Evaluate conditions action = evaluateConditions(step.conditions, result) // next | retry | finish | jump(stepId) // 6. Checkpoint saveCheckpoint(executionId, stepIndex, result) }
⚖

DAGWorkflowExecutor + DAGExecutor

dag-workflow-executor.ts + dag-executor.ts

Grafo Aciclico Dirigido com execucao em batches paralelos. O DAGExecutor gerencia o grafo de dependencias, detecta ciclos (DFS com sets visited + inStack com backtrack), valida referencias e orquestra batches. Steps independentes executam via Promise.all(). Input de cada step: userInput + '\n\n' + depContext onde depContext formata como [Output from "StepName"]:\n{content}. Falha parcial: se um step com handler='fail' falha, para IMEDIATAMENTE (outros do batch podem ainda estar rodando).

✓ validate(): verifica deps existem + detecta ciclos DFS
✓ getReadySteps(): retorna steps sem deps pendentes
✓ Execucao em batches paralelos (Promise.all)
✓ markCompleted(stepId, output): atualiza grafo
✓ getDependencyContext(stepId): outputs das deps
✓ Emite dag:batch_start por batch
✓ isComplete(): todos steps terminaram?
Exemplo de Execucao DAG:
Batch 1 (paralelo)
Step A (raiz) Step B (raiz)
↓ ambos completam
Batch 2 (paralelo)
Step C (depende A) Step D (depende B)
↓ ambos completam
Batch 3
Step E (depende C+D)
✋

StepByStepExecutor

step-by-step-executor.ts

Executa UM step por vez. O cliente controla o avanço via API. Cada step gera ExecutionState separado. O frontend mostra botao "Advance" no StepPanel.

6 Pipeline de Execucao do Step

Os 9 estagios que cada step percorre dentro do StepExecutor.executeStep()

Session Management

Recupera sessao existente via sessionManager.getSession(conversationId, stepId). Sessoes mantêm estado do Claude CLI entre retries. Se sessao nao existe, cria nova. Se isSessionNotFoundError() detectar "no conversation found" / "session not found" / "session_not_found", deleta sessao e retenta sem resume token.

Memory Management

Recupera memoria do step via memoryManager.getMemory(conversationId, stepId). StepMemory persiste contexto de conversacao entre compactacoes do Claude. Stored em tabela step_memories.

System Prompt Construction

buildSystemPrompt() monta o prompt combinando ate 3 partes com separador \n\n---\n\n: (1) Base Prompt (11 secoes: Identity, Communication, Infrastructure, Database, Stack, Security, Git, Testing, Documentation, Architecture, Finalization) + (2) Step System Prompt + (3) Memoria (como secao "MEMORIA DE CONTEXTO" com instrucao para continuar de onde parou).

File Synchronization

fileSyncService.syncForStep(projectPath, stepId) sincroniza arquivos do workspace para o step. Garante estado consistente do filesystem.

Git Configuration

Configura git user.name/email do banco de dados. Configura GitHub token se disponivel. So executa se projectPath e repositorio git.

Monitoring Setup

ExecutionMonitor rastreia metricas: stdout/stderr (50KB limit), parsedEvents, tokens, timing (startedAt, firstByteAt, firstContentAt, completedAt, durationMs).

CLI Engine Execution

engine.execute(options) executa o Claude CLI: sudo -u claude-sandbox claude --resume ${token} -p "${message}" --system-prompt "${prompt}". Encapsulado por withTimeout(promise, ms, name) que cria race condition: ou a promise resolve ou rejeita com Error("Step X excedeu timeout de Ys"). Se timeout dispara, chama engine.cancel() para matar o processo.

Result Handling

6 cenarios: Success (content + actions), Timeout (timedOut=true, engine.cancel()), Interrupt (salva conteudo parcial + mensagem interrupt, re-executa com interrupt message), SessionNotFound (deleta sessao, retenta sem resume), PromptTooLong (usa Claude Haiku para summarize memoria, rebuilda systemPrompt com memoria compactada, retenta sem resume), NeedsUserInput (pausa execucao).

Output Persistence

Cria Message record no banco com role, content, stepId, metadata (actions[], sessionId, stepName, stepOrder). Emite eventos message:saved e step:complete. Flush do ExecutionMonitor para ExecutionTrace via fire-and-forget com retry exponencial (1s, 4s, 9s). Registra token budget via tokenBudgetService.recordUsage().

Mecanismo de Retry

Quando uma condition resolve para retry:

  1. Verifica retry count do step vs maxRetries (rule.maxRetries → step.maxRetries → default 3)
  2. Se retries disponiveis: incrementa contador em retryCounts[step.id]
  3. Aplica backoffDelay(attempt): 2^attempt * 1000ms, cap em 30s (2s, 4s, 8s, 16s, 30s)
  4. Formata retryMessage com templates: {output}, {error} (=rule.match), {type} (=rule.type)
  5. Fallback se sem template: "A saida anterior nao atendeu aos criterios. Por favor, tente novamente."
  6. Re-executa o step com currentInput = retryMessage
  7. Se max retries atingido: emite stepComplete + avanca para proximo step
Backward Jumps: Quando condition faz jump para step anterior (nextIndex < currentIndex), e tratado como retry com contador separado: retryCounts[jump_${fromId}_to_${toIdx}]. Se max retries atingido, pula para frente em vez de criar loop infinito.
// Exemplo: retry com mensagem formatada { "rules": [{ "type": "not_contains", "match": "TESTS_PASSED", "goto": "retry", "maxRetries": 3, "retryMessage": "Os testes falharam. Output anterior: {output}. Corrija e tente novamente." }], "default": "next" }

8 Pause & Resume

Mecanismo de pausa via AskUserQuestion e retomada de execucao

Quando Pausa Ocorre

Quando o Claude CLI detecta a tool AskUserQuestion, o result retorna needsUserInput: true. O executor percorre result.actions[] buscando type === 'tool_use' && name === 'AskUserQuestion', extrai input.questions[0].question e options[]. Salva assistant message com flag needsUserInput: true + sessionId: resumeToken. Marca ExecutionState como paused e emite execution:paused.

interface PausedExecutionInfo { executionId: string conversationId: string stepIndex: number stepId: string resumeToken: string | null pausedAt: string // ISO timestamp askUserQuestion?: { question: string // Pergunta ao usuario options?: Array<{ label: string description?: string }> } }

Fluxo de Resume

1
API recebe resposta do usuario: POST /conversations/:id/resume ou conversation_send_message
2
ExecutionResumer recupera estado pausado via getPausedExecution(conversationId)
3
Verifica se resumeToken existe (continuidade de sessao)
4
Chama executeStep() novamente com: mesmo step + resposta do usuario + resumeToken original
5
Marca ExecutionState como running. Continua com proximo step normalmente.
!
Pode pausar novamente: Se durante o resume o step chamar AskUserQuestion outra vez, o fluxo de pausa se repete. O ExecutionResumer tambem continua executando todos os steps restantes do workflow sequencial apos o resume.

Fluxo de Versionamento

v1 — Criacao
Snapshot inicial ao criar workflow. Sem diff. workflowVersioningService.createVersion() e chamado automaticamente.
v2 — Edicao
PUT /workflows/:id trigga automaticamente novo snapshot via createVersion(). Diff calculado via computeDiff(prev, curr) que compara snapshots e retorna { added: string[], removed: string[], modified: string[] } de steps.
v3 — Rollback para v1
POST /workflows/:id/versions/rollback {version: 1}. Transacao atomica: (1) deleteStepsAndJoins() remove todas join tables (MCP, Skill, Agent, Rule, Hook) + steps, (2) Recria steps com nested creates das join tables, (3) Atualiza workflow (name, desc, type), (4) Cria v3 com changelog "Rollback para versao 1".

18 Workflow Triggers

5 tipos de trigger que disparam execucao automatica de workflows

✋ manual
Disparado manualmente via API ou UI. Padrao.
⏰ cron
Agendamento via expressao cron. Suporta timezone (cronTimezone). Ex: "0 9 * * 1-5" (9h seg-sex).

Disparado por webhook externo. Validacao via webhookSecret.
⚡ event
Disparado por evento interno (ex: workflow_complete). Filtros via eventFilter JSON.
📤 git_push
Disparado por push em repositorio git. Integrado com git hooks.

19 Lifecycle Completo de Execucao

Fluxo end-to-end desde a criacao ate o resultado final

1. CREATE WORKFLOW POST /workflows

workflowsRepository.create() → Cria Workflow + WorkflowSteps + Join Tables atomicamente → syncWorkflowDependencies() → workflowVersioningService.createVersion(v1)

2. START EXECUTION POST /api/v1/workflows/:id/execute

tokenBudgetService.checkBudget(userId) → getOrCreateConversation() → projectPathLock.acquire(projectPath)

3. ROUTING TaskOrchestrator

Analisa type + dependsOn → Roteia para SequentialExecutor / DAGWorkflowExecutor / StepByStepExecutor → Cria ExecutionState(running)

4. STEP EXECUTION LOOP StepExecutor.executeStep()

Session → Memory → SystemPrompt → FileSync → Git → Monitor → engine.execute() → Result → Persist → Validators → Conditions → Next step

5. COMPLETION collectExecutionResult()

Mark ExecutionState(completed) → finalizeExecution(): extractToolCalls + createOrUpdateAggregate → Emit execution:complete → webhooksService.dispatch('execution:complete') → projectPathLock.release() → collectExecutionResult(): gather all assistant Messages by step

ALT: ERROR PATH

Se step falha: StepErrorHandler.decideOnError() → fail (markFailed + return) | skip (i++, continue) | fallback (re-executa com "Retry: a execucao anterior falhou. Simplifique.") | continue_next (ignora erro). Catch global: markFailed + finalizeExecution + webhooksService.dispatch('step:error'). Finally: SEMPRE deleta activeExecutions[conversationId].

ALT: PAUSE PATH

Se Claude chama AskUserQuestion: extrai question + options do actions[] → salva assistant message com needsUserInput: true → markPaused(stepIndex, stepId, resumeToken, askQuestion) → Emit execution:paused → delete activeExecutions → return. Resume: usuario responde → ExecutionResumer.resumeExecution() → executeStep() com mesmo step + resposta + resumeToken → continua sequencial com steps restantes. Pode pausar NOVAMENTE.
