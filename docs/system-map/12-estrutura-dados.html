1. Queries Perigosas & Performance

14 repositories com findMany sem paginacao, 3 full-table scans, 4 operacoes sem $transaction

CRITICO
AS-IS Encontrado no Codigo
Full-table scans em metricas:
  • metrics.routes.ts:15 — /metrics/daily carrega TODOS traces de 30 dias em memoria para agregar em JS
  • metrics.routes.ts:51 — /metrics/by-workflow carrega TODOS traces sem limite temporal para agregar por workflow
  • metrics.routes.ts:93 — /metrics/alerts carrega 24h de traces para checagem de threshold
Operacoes sem $transaction:

  • plugins.repository.ts:223-227 — Delete cascata: 4 tabelas sem transaction

findMany sem paginacao (14 arquivos):
conversations, workflows, skills, agents, mcp-servers, plugins, hooks, rules, webhooks, step-templates, api-keys, admin (users), metrics (traces)

3. Motor de Execucao — Gaps de Dados

errorCategory nunca populado, custo USD do budget nao checado, trace cleanup destrutivo

6 GAPS
AS-IS
errorCategory nunca populado: O enum ErrorCategory existe no schema, mas execution-monitor.ts flush() nunca classifica o erro. Todos os traces ficam com errorCategory = null.
Budget em USD nao checado: Campos dailyCostLimitUsd e monthlyCostLimitUsd existem no schema, mas checkBudget() so verifica tokens, nao custo.
Trace cleanup destrutivo: trace-cleanup.ts faz deleteMany de traces com 7+ dias sem agregar antes. Dados historicos perdidos para sempre.
Fire-and-forget no flush: execution-monitor.ts:155-212 — persistencia do trace e .catch(() => {}). Se falhar, trace perdido sem retry.
Cache efficiency nao calculada: Tokens de cache (creation/read) sao gravados, mas nenhuma metrica de "cache hit rate" e derivada.
TTFT nao exposto: firstByteAt e firstContentAt gravados no trace, mas nao calculados nem expostos em nenhum endpoint.
TO-BE
// 1. Classificar erro no flush()
function classifyError(exitCode, errorMsg, stopReason): ErrorCategory {
  if (errorMsg?.includes('timeout')) return 'timeout'
  if (errorMsg?.includes('rate limit')) return 'rate_limit'
  if (exitCode === 137) return 'process_killed'
  if (stopReason === 'max_tokens') return 'context_overflow'
  return 'unknown'
}

// 2. Checar custo no budget
if (budget.dailyCostLimitUsd && currentCost >= budget.dailyCostLimitUsd) {
  throw new BudgetExceededError('Limite diario de custo atingido')
}

// 3. Agregar ANTES de deletar traces
async function cleanupOldTraces() {
  await aggregateIntoDaily(tracesOlderThan7Days) // primeiro agrega
  await prisma.executionTrace.deleteMany(...)     // depois deleta
}

// 4. Retry no flush com dead-letter
try { await persistTrace(data) }
catch { await deadLetterQueue.add(data) } // nao perde

4. Workflows — Campos Fantasma & Logica Faltante

subWorkflowId, skipCondition, outputSchema existem no step mas nao sao avaliados pelo orchestrator

5 FANTASMAS
FANTASMA
subWorkflowId: Campo existe no WorkflowStep, mas TaskOrchestrator nao tem logica para executar sub-workflows. Campo morto.
skipCondition: Campo existe, mas orchestrator nao avalia antes de executar o step.
outputSchema: Campo existe, mas orchestrator usa validators separado — outputSchema e ignorado.
WorkflowVersion: Model existe, mas workflow update ainda faz DELETE ALL + CREATE sem criar snapshot.
Retry sem backoff: maxRetries funciona, mas retry e imediato — sem delay/exponential backoff.
AS-IS
Race condition no step navigation: advanceStep(), goBack(), jumpToStep() leem currentStepIndex, calculam proximo, e gravam sem locking. Requests concorrentes podem pular steps.
Sem budget por workflow: Budget e global por usuario. Nao ha como limitar "esta conversa pode gastar no maximo $5".
TO-BE
1. Implementar avaliacao de skipCondition no orchestrator antes de executar step
2. Criar snapshot em WorkflowVersion antes de DELETE ALL steps
3. Adicionar delay entre retries: await sleep(retryDelay * 2^attempt)
4. Usar $transaction com locking otimista no advanceStep para evitar race condition
5. Implementar sub-workflow execution no orchestrator (compor workflows como blocos)

5. Recursos — Dependencias Quebradas

ResourceDependency nao populado, Agent.skills duplicado, Git tracking 3x repetido

ALTO
AS-IS
Dependencias invisiveis: ResourceDependency existe no schema mas nenhum service cria registros. Deletar MCP server ainda quebra skills silenciosamente via onDelete: Cascade nas join tables.
Agent.skills duplicado: Campo skills String @default("[]") no Agent guarda IDs como JSON. Mas existe AgentSkill join table. Duas fontes de verdade — qual e a correta?
Git tracking 3x: Campos repoOwner, repoName, repoBranch, repoPath, lastSyncedAt duplicados em Skill, Agent e Rule. GitSource model existe mas nao e usado.
Ownership nao verificada: update() em skills, agents, hooks, rules faz findUnique({where: {id}}) sem checar userId. Qualquer usuario que conheca o ID pode editar recurso de outro.
TO-BE
1. Popular ResourceDependency ao vincular recursos a steps (hook no service)
2. Remover Agent.skills String e usar apenas a join table AgentSkill
3. Migrar campos git inline para GitSource (ja existe no schema)
4. Adicionar where: { id, userId } em todos os findUnique de update/delete

6. Seguranca — Tokens em Texto Puro

GitAccount.token, Webhook.secret, McpServer.envVars sem criptografia

CRITICO
AS-IS
Tokens GitHub em plaintext: GitAccount.token armazenado sem criptografia. Se o banco vazar, todos os tokens sao expostos.
Webhook secrets em plaintext: Webhook.secret e WorkflowTrigger.webhookSecret idem.
MCP envVars podem conter secrets: McpServer.envVars e JSON com possiveis API keys e senhas.
Auth nao valida que user existe: JWT path decodifica token e retorna sub, mas nao verifica se o user ainda existe no banco (pode ter sido deletado).
Seed com senha hardcoded: prisma/seed.ts usa admin123. Pode ser rodado em producao.
TO-BE
// Criptografar tokens com AES-256-GCM
import { createCipheriv, createDecipheriv, randomBytes } from 'crypto'

function encrypt(text: string): string {
  const iv = randomBytes(16)
  const cipher = createCipheriv('aes-256-gcm', ENCRYPTION_KEY, iv)
  const encrypted = Buffer.concat([cipher.update(text), cipher.final()])
  const tag = cipher.getAuthTag()
  return iv.toString('hex') + ':' + tag.toString('hex') + ':' + encrypted.toString('hex')
}

7. Frontend — Dados que o Backend Tem mas o Frontend Nao Consome

Dashboard incompleto, sem token budget viz, sem tool analytics, ExecutionTrace rico mas invisivel

10 GAPS
AS-IS
Dashboard simplificado demais: Backend tem DailyExecutionMetrics com p50/p95, model breakdown, error categories. Frontend so consome date, cost, tokens, executions, errors — sem latencia, sem breakdown.
Token budget invisivel: UserTokenBudget com daily/monthly limits existe. Frontend nao tem nenhum componente mostrando progresso do budget.
Tool analytics zero: ExecutionToolCall grava dados no backend. Nenhum endpoint ou componente mostra quais tools sao mais usadas.
ExecutionTrace rico mas oculto: 46 campos (stdoutRaw, stderrRaw, parsedEvents, iterations, modelUsage, permissionDenials). Frontend nao acessa nenhum desses diretamente.
Custo nao aparece nas listas: Conversation tem totalCostUsd, mas a lista de conversas nao retorna esse campo.
Sem indicador de execucao ativa na lista: Precisa buscar ExecutionState separadamente para cada conversa.
Sem busca full-text em mensagens: Busca so por title e workflowName.
Sem timeline de execucao: parsedEvents tem timeline compacta, mas nao ha "waterfall view" no frontend.
Workflow sem lastRunAt: Impossivel saber quando o workflow rodou por ultimo sem navegar para execucoes.
Stats cards sem trend: Mostram numeros absolutos ("5 workflows") sem delta vs periodo anterior.
TO-BE
1. Criar endpoint /metrics/dashboard que retorne dados completos do DailyExecutionMetrics (p50/p95, breakdown)
2. Criar componente TokenBudgetBar que mostre uso diario/mensal vs limite
3. Criar endpoint /metrics/tools com top tools, success rate, avg duration
4. Retornar totalCostUsd e status na listagem de conversas
5. Adicionar lastExecutedAt e executionCount na resposta de listagem de workflows
6. Criar componente ExecutionTimeline que renderize parsedEvents como waterfall
