1. Autenticacao & Identidade

User, ApiKey, UserTokenBudget — 3 modelos

3 FKs orfas +12 campos
AS-IS Estado Atual
User: Modelo basico com githubToken legado (duplicado com GitAccount), role como String livre sem validacao, sem campos de perfil, status ou preferencias.
ApiKey: Sem scopes (toda chave tem acesso total), sem rate limit, sem descricao do proposito.
TokenBudget: Apenas limites em tokens brutos. Reset por logica manual com race condition em operacoes concorrentes. Sem historico — dados perdidos a cada reset.
// User — sem status, sem preferences, role sem validacao
model User {
  role        String  @default("developer")  // qualquer string aceita
  githubToken String? // legado, texto puro
}

// ApiKey — acesso total, sem scopes
model ApiKey {
  keyHash  String @unique
  // sem scopes, sem rateLimit, sem usageCount
}

// Budget — race condition no recordUsage
async recordUsage(userId, tokens) {
  const budget = await this.getOrCreate(userId) // pode resetar
  await prisma.update({ increment: tokens })    // outra execucao pode ter resetado entre estas linhas
}
TO-BE Estado Proposto
User: Campo status (active/suspended), role como enum, preferences JSON, lastLoginAt, maxConcurrentExecutions, remocao do githubToken legado.
ApiKey: scopes JSON (read/write/execute), rateLimit, usageCount, ipWhitelist.
TokenBudget: Limites em USD alem de tokens, alertThreshold (80%), overagePolicy (block/warn/allow), historico via TokenUsageHistory.
// User — completo com enum e campos de gestao
enum UserRole { admin developer viewer }
enum UserStatus { active suspended invited }

model User {
  role            UserRole   @default(developer)
  status          UserStatus @default(active)
  lastLoginAt     DateTime?
  preferences     Json       @default("{}")
  maxConcurrent   Int        @default(3)
  // githubToken REMOVIDO — usar GitAccount
}

// ApiKey — granular com scopes
model ApiKey {
  scopes      Json     @default("[\"read\"]")
  rateLimit   Int?     // requests/min
  usageCount  Int      @default(0)
  ipWhitelist String?  // JSON array de IPs permitidos
}

// Budget — com historico e USD
model UserTokenBudget {
  dailyCostLimitUsd    Float?
  monthlyCostLimitUsd  Float?
  alertThreshold       Float  @default(0.8)
  overagePolicy        String @default("block")
}

// NOVO: Historico de consumo
model TokenUsageHistory {
  id        String   @id @default(uuid())
  userId    String
  date      DateTime @db.Date
  tokens    BigInt
  costUsd   Float
  @@unique([userId, date])
}

2. Workflows & Steps

Workflow, WorkflowStep, StepTemplate, PromptVersion — 4 modelos

Sem versionamento +20 campos
AS-IS Problemas Criticos
Sem versionamento: Ao editar um workflow, o repository faz DELETE ALL steps + CREATE ALL, destruindo todo historico. Sem rollback possivel.
Sem status/draft: Todo workflow criado esta imediatamente "ativo". Nao ha conceito de rascunho, arquivo ou publicacao.
JSON como String: conditions, dependsOn, validators, inputVariables, outputVariables — 5 campos JSON serializados como String.
StepTemplate desconectado: userId sem relacao com User, templateId no WorkflowStep sem FK. Nenhuma integridade referencial.
Sem timeout/error handling: Se um step trava, nao ha como abortar. Se falha, todo o workflow para.
// Update workflow = DESTRUIR + RECRIAR todos os steps
await prisma.workflowStep.deleteMany({ where: { workflowId } })
await prisma.workflowStep.createMany({ data: newSteps })
// versao anterior? PERDIDA PARA SEMPRE

// WorkflowStep — 5 campos JSON como String
conditions      String @default("[]")   // JSON.parse() obrigatorio
dependsOn       String @default("[]")   // sem query possivel
validators      String @default("[]")   // sem validacao DB
outputVariables String @default("[]")   // sem indexacao
inputVariables  String @default("[]")   // sem tipo
TO-BE Proposta Completa
Versionamento: Novo modelo WorkflowVersion com snapshot completo, diff estrutural e rollback.
Status: Campo status (draft/active/archived) + lastExecutedAt + executionCount.
DAG avancado: timeout, errorHandler (fail/skip/fallback/continue), skipCondition, subWorkflowId, outputSchema.
// NOVO: Versionamento completo
model WorkflowVersion {
  id            String   @id @default(uuid())
  workflowId    String
  version       Int
  label         String?  // "v2.1 - added QA step"
  snapshot      Json     // workflow + steps + resources
  diff          Json?    // diff vs versao anterior
  changeType    String   // step_added, prompt_changed...
  isPublished   Boolean  @default(false)
  workflow      Workflow @relation(...)
  @@index([workflowId, version])
}

// Workflow com status e metricas
model Workflow {
  status          String  @default("active") // draft/active/archived
  tags            Json    @default("[]")
  lastExecutedAt  DateTime?
  executionCount  Int     @default(0)
  timeoutMs       Int?
  versions        WorkflowVersion[]
}

// WorkflowStep com campos avancados
model WorkflowStep {
  conditions    Json    @default("[]") // tipo nativo
  dependsOn     Json    @default("[]")
  timeout       Int?    // ms, abort se exceder
  errorHandler  String  @default("fail") // fail|skip|fallback|continue
  fallbackStepId String?
  skipCondition String? // expression para pular step
  subWorkflowId String? // composicao!
  outputSchema  Json?   // JSON Schema para validar output
  description   String?
  updatedAt     DateTime @updatedAt
}

3. Motor de Execucao

ExecutionState, ExecutionLog, ExecutionTrace — 3 modelos, 46+ campos

Sem agregacao Full-table scans
AS-IS Gargalos Graves
Sem agregacao: /metrics/daily e /metrics/by-workflow fazem scan completo de execution_traces e agregam em JS. Nao escala.
Sem error taxonomy: resultStatus e String livre ("success"/"error"). Sem categorizacao (timeout, rate_limit, context_overflow, budget_exceeded).
Tool usage invisivel: Uso de ferramentas esta dentro de parsedEvents JSON opaco. Impossivel saber "qual tool e mais usada?"
Cleanup destrutivo: Traces deletados apos 7 dias sem agregacao previa. Dados historicos perdidos para sempre.
3 FKs orfas: stepId, executionId, promptVersionId no ExecutionTrace sao Strings sem integridade referencial.
// metrics.routes.ts — full table scan em TODA chamada
const traces = await prisma.executionTrace.findMany({
  where: { conversation: { userId } }  // TODOS os traces do usuario
})
const daily = {} // agrega em JS...
traces.forEach(t => {
  const day = t.createdAt.toISOString().split('T')[0]
  daily[day] = (daily[day] || 0) + t.totalCostUsd
})

// Cleanup — deleta sem salvar nada
await prisma.executionTrace.deleteMany({
  where: { createdAt: { lt: sevenDaysAgo } }
}) // historico? adeus
TO-BE Observabilidade Completa
3 camadas: DailyExecutionMetrics (pre-agregado), ExecutionAggregate (por execucao), ExecutionToolCall (por ferramenta).
Error taxonomy: errorCategory enum (timeout, rate_limit, context_overflow, budget_exceeded, tool_error, permission_denied).
Retencao inteligente: 7 dias raw traces, 90 dias agregados diarios, indefinido mensal.
// NOVO: Metricas pre-agregadas — ZERO full-table scans
model DailyExecutionMetrics {
  date             DateTime @db.Date
  userId           String
  workflowId       String?
  model            String?
  executionCount   Int      @default(0)
  successCount     Int      @default(0)
  errorCount       Int      @default(0)
  totalCostUsd     Float    @default(0)
  totalInputTokens BigInt   @default(0)
  durationP50Ms    Int?
  durationP95Ms    Int?
  toolUsageCounts  Json     @default("{}")  // {"Bash":45,"Read":30}
  errorCategories  Json     @default("{}")  // {"timeout":2}
  @@unique([date, userId, workflowId, model])
}

// NOVO: Agregado por execucao (cross-step)
model ExecutionAggregate {
  id              String @id
  conversationId  String
  workflowId      String
  totalSteps      Int
  completedSteps  Int
  failedSteps     Int
  totalCostUsd    Float  @default(0)
  totalDurationMs Int?
  finalStatus     String
  errorCategory   String?
}

// NOVO: Tool calls extraidos do JSON opaco
model ExecutionToolCall {
  id         String  @id @default(uuid())
  traceId    String
  toolName   String  // Bash, Read, Edit, Write...
  durationMs Int?
  success    Boolean @default(true)
  @@index([toolName])
  @@index([traceId])
}

// ExecutionTrace com FKs reais e error taxonomy
model ExecutionTrace {
  step           WorkflowStep? @relation(...)  // FK real
  execution      ExecutionState @relation(...)  // FK real
  errorCategory  String?  // timeout|rate_limit|...
  errorCode      String?  // machine-readable
  isRetryable    Boolean?
  toolCalls      ExecutionToolCall[]
}

4. Conversas & Mensagens

Conversation, Message, MessageAttachment, Session, StepMemory — 5 modelos

Sem status +15 campos
AS-IS
Conversation sem status: Para saber se esta "rodando", precisa checar taskOrchestrator.isExecuting(id) em memoria + executionState. Fragil.
Sem custo/duracao na listagem: totalCost e duration so disponiveis via endpoint separado, impossivel mostrar na lista.
Message sem tokenCount: Impossivel saber quanto cada mensagem custou.
findById carrega TUDO: include: { messages: { include: { attachments } } } — todas as mensagens de uma vez. Em conversas longas = memoria explodindo.
Sem busca full-text: Busca em message.content faria scan completo sem indice.
TO-BE 
// Conversation com status e metricas denormalizadas
model Conversation {
  status          String    @default("active") // active|completed|failed|archived
  completedAt     DateTime?
  totalTokensUsed Int       @default(0) // cache para listagem rapida
  totalCostUsd    Float     @default(0)
  lastMessageAt   DateTime?
  isArchived      Boolean   @default(false) // soft delete
  tags            Json      @default("[]")
  parentId        String?   // fork de conversa
}

// Message com metricas
model Message {
  role       String // enum: user|assistant|system
  tokenCount Int?   // custo da mensagem
  model      String? // qual modelo gerou
  metadata   Json?   // tipo nativo, nao String
}

// NOVO: Feedback por mensagem
model MessageFeedback {

5. Recursos (MCP, Skills, Agents, Rules, Plugins, Hooks)

6 modelos + 5 join tables — mais complexo dominio do sistema

Deps quebradas GitHub 3x duplicado
AS-IS
Dependencias invisiveis: Skill.allowedTools referencia tools por nome (String), nao por ID de MCP. Deletar um MCP quebra Skills silenciosamente.
Agent.skills duplicado: Campo skills String @default("[]") guarda IDs de Skills como JSON. Mas ja existe join table WorkflowStepSkill. Duas fontes de verdade.
Git tracking 3x: Campos repoOwner, repoName, repoBranch, repoPath, lastSyncedAt duplicados em Skill, Agent e Rule.
Sem tags/categorias: Nenhum recurso tem campo de tag ou categoria. Impossivel organizar com dezenas de Skills.
Join tables sem metadata: Nenhuma join table tem createdAt, order ou config. Impossivel saber quando foi vinculado ou customizar por step.
Hook fora do Plugin: Unico recurso sem pluginId. Inconsistencia arquitetural.
TO-BE 
// NOVO: Dependencias declarativas entre recursos
model ResourceDependency {
  id             String @id @default(uuid())
  sourceType     String // "skill"|"agent"|"rule"|"hook"
  sourceId       String
  dependencyType String // "requires_mcp"|"requires_skill"
  dependencyRef  String // ID do recurso dependente
  isOptional     Boolean @default(false)
  @@unique([sourceType, sourceId, dependencyType, dependencyRef])
}

// NOVO: Git source extraido (elimina duplicacao 3x)
model GitSource {
  id           String @id @default(uuid())
  resourceType String // "skill"|"agent"|"rule"
  resourceId   String
  repoOwner    String
  repoName     String
  repoBranch   String @default("main")
  repoPath     String?
  lastSyncedAt DateTime?
  @@unique([resourceType, resourceId])
}

// NOVO: Tags universais

// Join tables enriquecidas

6. Observabilidade & Metricas

0 modelos dedicados — tudo calculado em tempo real com full scans

+5 novos modelos
AS-IS Nenhum modelo de observabilidade
Metricas calculadas on-the-fly via full scan de ExecutionTrace. Sem percentis (p50/p95), sem cost breakdown, sem trend data, sem tool analytics.
Dashboard mostra numeros sem contexto — "5 workflows" sem delta vs ontem. TokenBudget existe mas nao aparece no frontend.
Perguntas impossiveis: "Qual e o custo medio por workflow?" "Qual tool e mais lenta?" "A qualidade melhorou apos mudar o prompt?"
