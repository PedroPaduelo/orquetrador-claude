Problemas Corrigidos (6/14)

CORRIGIDO

Full Table Scans em Metrics

ANTES: 3 endpoints carregavam TODAS as traces em memoria para agregar em JS. /metrics/daily carregava 30 dias, /metrics/by-workflow nao tinha filtro temporal.

AGORA:

  • /metrics/daily usa DailyExecutionMetrics pre-agregada com filtro de data
  • /metrics/by-workflow usa groupBy() com agregacao no banco
  • /metrics/alerts usa aggregate() + count() com filtro 24h

metrics.routes.ts — Zero full table scans

CORRIGIDO

$transaction em Workflows (update de steps)

ANTES: 7 operacoes sequenciais (delete join tables + delete steps + create novos) sem transacao. Falha no meio = workflow corrompido.

AGORA: Toda a operacao envolvida em prisma.$transaction(async (tx) => { ... }). As 7 operacoes sao atomicas.

workflows.repository.ts:249-305

CORRIGIDO

$transaction em Plugins (delete cascata)

ANTES: 4 deletes sequenciais (mcpServer → skill → agent → plugin) sem transacao.

AGORA: Envolvido em prisma.$transaction([...]) com as 4 operacoes atomicas.

plugins.repository.ts:222-229

CORRIGIDO

Race Condition no Token Budget

ANTES: checkBudget() e recordUsage() nao-atomicos. Dois requests simultaneos podiam estourar o limite.

AGORA: recordUsage() envolvido em prisma.$transaction(). Reset de periodo e incremento atomicos. dailyLimit e monthlyLimit agora sao verificados.

token-budget-service.ts:59-75

CORRIGIDO

errorCategory Nunca Populada

ANTES: Campo errorCategory no ExecutionTrace nunca era classificado. Error breakdown no dashboard sempre vazio.

AGORA: Funcao classifyError() analisa resultStatus, exitCode, errorMessage e stopReason para classificar em: timeout, rate_limit, context_overflow, budget_exceeded, tool_error, permission_denied. Chamada automaticamente no flush().

execution-monitor.ts:25-45

CORRIGIDO

Git Accounts N+1 (HTTP sequencial)

ANTES: Para cada conta sem username cached, fazia uma chamada HTTP sequencial ao GitHub + update no banco. 10 contas = 10 calls sequenciais.

AGORA: Usa Promise.all() para validacao paralela. Campo username cacheado evita validacoes repetidas.

git.routes.ts:250-273

Parcialmente Corrigidos (2)

PARCIAL

Webhook Ownership (update/delete)

O QUE MUDOU: Agora faz findFirst({ where: { id, userId } }) para verificar ownership antes de operar.

O QUE FALTA: O update() e delete() subsequentes usam apenas where: { id } sem re-validar userId. Correto seria where: { id, userId } na operacao final, ou fazer tudo em $transaction.

webhooks.repository.ts:47-63

PARCIAL

Fire-and-forget Traces (sem retry)

O QUE MUDOU: Error handling com logging do erro.

O QUE FALTA: Se prisma.executionTrace.create() falhar, a trace e perdida permanentemente. Nao ha retry automatico nem dead-letter queue. O mesmo vale para createOrUpdateAggregate e extractToolCallsForExecution.

execution-monitor.ts:177-244

Ainda Pendentes (6 criticos)

PENDENTE

Zero Paginacao em 9 Repositories

TODOS os findAll() carregam TODOS os registros do usuario sem take/skip:

conversations workflows skills agents mcp-servers plugins hooks rules webhooks

Impacto: Com centenas de registros, cada listagem carrega tudo em memoria. Problema de escala.

Fix: Adicionar parametros page + pageSize em cada findAll() com take/skip + retornar total via count().

PENDENTE

Tokens Senssiveis em Plaintext

GitAccount.token e Webhook.secret armazenados sem criptografia. Se o banco vazar, todos os tokens GitHub ficam expostos.

Fix: Criptografar com AES-256-GCM usando chave do environment. Decrypt apenas no momento do uso. Middleware Prisma pode automatizar.

PENDENTE

JWT Nao Valida Existencia do User

O path JWT decodifica o token e retorna o sub (userId) sem verificar se o usuario ainda existe no banco. Se um usuario for deletado, seu JWT antigo continua funcionando.

Nota: O path de API Key FAZ a verificacao. Apenas JWT esta inconsistente.

Fix: Adicionar prisma.user.findUnique({ where: { id: sub } }) no auth JWT. Cache em Redis (TTL 5min) para nao bater no BD a cada request.

auth.ts:72-78

PENDENTE

Race Condition na Navegacao de Steps

advanceStep(), goBack() e jumpToStep() fazem Read → Logic → Write sem locking. Dois requests simultaneos podem ler o mesmo currentStepIndex e corromper a navegacao.

Fix: Usar prisma.$transaction com serializable isolation level, ou optimistic locking com campo version.

conversations.service.ts:8-100

PENDENTE

$transaction em Conversations (create/clone)

create() e clone() fazem findUnique do workflow e depois create da conversa sem transacao. Se o workflow for deletado entre as duas queries, cria conversa orfa.

conversations.repository.ts:103-199

PENDENTE

Zero Prisma Migrations

Diretorio prisma/migrations/ nao existe. Schema aplicado com prisma db push que pode causar perda de dados em producao.

Fix: npx prisma migrate dev --name init para criar migration base. Depois usar prisma migrate deploy em producao.

PENDENTE

CORS Permissivo (origin: true)

origin: true aceita qualquer origin, permitindo CSRF. Para single-user e toleravel, mas arriscado com app publico.

server.ts:64

PENDENTE

SIGINT/SIGTERM chama process.exit(0) sem aguardar execucoes ativas terminarem. Processos Claude CLI ficam orfaos.

server.ts:211-223

PENDENTE

Recovery sobrescreve metadata

No restart, recovery.ts faz JSON.stringify({ error, failedAt }) que SOBRESCREVE toda a metadata existente da execucao em vez de fazer merge.

recovery.ts:8-12
