1 3 Niveis de Error Handling

O Orquetrador trata erros em 3 camadas distintas. Cada nivel captura falhas que o nivel anterior nao tratou.

1
HTTP Level
Global Error Handler

Intercepta erros em TODAS as requisicoes HTTP antes de enviar a resposta.


Arquivo: error-handler.ts

Responsabilidade:
  • Mapear erros custom para status HTTP
  • Formatar erros Zod para o cliente
  • Tratar erros do Prisma/Redis
  • Ocultar stack traces em producao
2
Step Level
Step Error Handler

Cada step de um workflow tem um errorHandler configuravel que decide o que fazer quando o step falha.


Arquivo: step-error-handler.ts

Opcoes:
  • fail — para tudo
  • skip — pula o step
  • fallback — tenta alternativa
  • continue_next — segue em frente
3
Execution Level
Recovery

Na inicializacao do servidor, detecta execucoes orfas (estado running sem processo ativo) e marca como failed.


Arquivo: recovery.ts

Mecanismo:
  • Query stale em ExecutionState
  • Marca como failed com metadata
  • Preserva metadata original
  • Log de quantas foram recuperadas

Classes de erro customizadas em src/http/errors/. Cada uma mapeia para um status HTTP especifico.

BadRequestError 400
bad-request-error.ts

Dados invalidos, campos obrigatorios faltando, formato incorreto. Usado em validacoes manuais.

UnauthorizedError 401
unauthorized-error.ts

Token JWT ausente, expirado ou invalido. Dispara auto-logout no frontend.

ForbiddenError 403
forbidden-error.ts

Autenticado mas sem permissao. Usado no middleware RBAC para roles insuficientes.

NotFoundError 404

Recurso nao encontrado. Tambem mapeado automaticamente de Prisma P2025.

ConflictError 409
conflict-error.ts

Conflito de unicidade (email duplicado, slug ja existente). Mapeado de Prisma P2002.

4 Error Categories (Prisma Enum)

O enum ErrorCategory no schema Prisma classifica erros de execucao. A funcao classifyError() em execution-monitor.ts analisa o resultado, exit code e mensagem para determinar a categoria.

⏱
timeout
Critico

Step excedeu o timeout (padrao: 5min). Tambem detectado se exitCode=137 (OOM/SIGKILL).

Match: "timeout" | "excedeu o timeout" | exitCode 137
🚦
rate_limit
Critico

API retornou 429 — limite de requisicoes por minuto atingido na API da Anthropic.

Match: "rate limit" | "rate_limit" | "429"
📏
context_overflow
Alto

Prompt longo demais para o modelo. Pode ser triggered por stop_reason=max_tokens.

Match: "prompt is too long" | "context_length" | stopReason="max_tokens"
💰
budget_exceeded
Alto

Orcamento de tokens diario ou mensal do usuario foi excedido.

Match: "budget" | "limite diario" | "limite mensal"
🔧
tool_error
Medio

Falha em um MCP server ou ferramenta. Geralmente transiente — pode ser retentado.

Match: "tool" + ("error" | "failed")
🔒
permission_denied
Medio

Acesso negado — permissao do sandbox, filesystem ou Claude CLI bloqueou a acao.

Match: "permission" | "forbidden" | "403"
❓
unknown
Baixo

Erro que nao se encaixa em nenhuma categoria acima. Usado como fallback para error e cancelled.

Match: resultStatus="error" | "cancelled" (sem match anterior)
classifyError() — Logica de Classificacao
export function classifyError( resultStatus: string, exitCode?: number | null, errorMessage?: string, stopReason?: string, ): ErrorCategory | null { // Resultados ok nao sao classificados if (resultStatus === 'success' || 'needs_input' || 'interrupted') return null const msg = errorMessage?.toLowerCase() ?? '' if (msg.includes('timeout')) → ErrorCategory.timeout if (msg.includes('rate limit')) → ErrorCategory.rate_limit if (msg.includes('prompt is too long')) → ErrorCategory.context_overflow if (msg.includes('budget')) → ErrorCategory.budget_exceeded if (msg.includes('permission')) → ErrorCategory.permission_denied if (msg.includes('tool') + 'error') → ErrorCategory.tool_error if (exitCode === 137) → ErrorCategory.timeout // OOM / SIGKILL return ErrorCategory.unknown // fallback }

5 Step Error Handlers (StepErrorHandler Enum)

Cada WorkflowStep tem um campo errorHandler (enum no Prisma) que define o comportamento quando o step falha. A funcao decideOnError() retorna um StepErrorDecision com as flags de acao.

✋
fail Default

Para a execucao INTEIRA do workflow. Marca como failed.

skipStep: false continueAsSuccess: false shouldRetryWithFallback: false
⏭
skip Tolerante

Pula este step e continua para o proximo. O step fica marcado como erro, mas o workflow segue.

skipStep: true continueAsSuccess: false shouldRetryWithFallback: false
🔄
fallback Retry

Tenta novamente com prompt simplificado (FALLBACK_MESSAGE). Ultima tentativa antes de falhar.

skipStep: false continueAsSuccess: false shouldRetryWithFallback: true
FALLBACK_MESSAGE = "Retry: a execucao anterior falhou. Tente novamente de forma mais simples."
▶
continue_next Resiliente

Continua para o proximo step como se tivesse tido sucesso. Util para steps opcionais.

skipStep: false continueAsSuccess: true shouldRetryWithFallback: false
StepErrorDecision — Interface de Retorno
export interface StepErrorDecision { action: 'fail' | 'skip' | 'continue_next' | 'fallback' skipStep: boolean // true = pula este step continueAsSuccess: boolean // true = trata como sucesso shouldRetryWithFallback: boolean // true = tenta com FALLBACK_MESSAGE logMessage: string // mensagem descritiva para log }

7 Retry Logic

Mecanismos de retry em diferentes camadas do sistema.

🔄
Step Fallback Retry

Quando o handler e fallback, o TaskOrchestrator re-executa o step com a mensagem FALLBACK_MESSAGE como prompt simplificado.


const FALLBACK_MESSAGE = 'Retry: a execucao anterior falhou. Tente novamente de forma mais simples.'

O retry acontece UMA vez. Se o fallback tambem falhar, o step e marcado como failed.

📊
Trace Persist Retry

O ExecutionMonitor tenta persistir o trace 3 vezes com backoff exponencial se o Prisma falhar.


1s
1o retry
4s
2o retry
9s
3o retry

Formula: attempt * attempt * 1000 ms (backoff quadratico).

🌐
React Query Retry

No frontend, o TanStack React Query esta configurado com retry: 1 para queries que falham.


Isso significa que cada query HTTP que falhar sera retentada automaticamente UMA vez antes de mostrar erro ao usuario.

🏷
retryCounts no ExecutionState

O campo retryCounts (JSON) no ExecutionState armazena quantas vezes cada step foi retentado.


// Formato do JSON: { "step-abc-123": 1, "step-def-456": 2 }

Permite tracking de retries por step ao longo de uma execucao.

8 Recovery from Crashes

O arquivo recovery.ts implementa deteccao e limpeza de execucoes orfas na inicializacao do servidor.

1. Servidor reinicia (startup)
O servidor foi reiniciado (crash, deploy, ou reinicio manual). A funcao recoverStaleExecutions() e chamada durante o boot.
2. Query stale executions
Busca no banco todos os ExecutionState com state = 'running'. Se o servidor morreu, essas execucoes nao tem mais processo ativo — sao orfas.
const staleExecutions = await prisma.executionState.findMany({ where: { state: 'running' }, select: { id: true, metadata: true } })
3. Marca como failed
Cada execucao orfa e atualizada para state: 'failed'. A metadata original e PRESERVADA, com campos adicionais:
metadata: { ...existingMeta, // metadata original preservada recoveryError: 'Server restarted during execution', recoveryFailedAt: '2026-03-24T10:30:00.000Z' }
4. Log e continua
Loga quantas execucoes foram recuperadas: [Recovery] Marked N stale executions as failed. Se o query ou update falhar, loga o erro mas NAO impede o servidor de iniciar.
⚠ Importante
O recovery atual marca execucoes orfas como failed — nao tenta re-executar. Isso previne loops infinitos de retry e garante que o estado do banco e consistente. O usuario pode re-disparar a execucao manualmente se necessario.

9 Frontend Error Handling

4 camadas de protecao no frontend React, desde o interceptor HTTP ate o error boundary.

1
Axios Interceptor — Auto Logout
api-client.ts: intercepta TODA resposta. Se status === 401, remove o token do localStorage e redireciona para /login (exceto se ja esta em /login ou /register).
401 → logout
2
React Query — Retry Automatico
TanStack React Query com retry: 1. Queries que falham sao retentadas uma vez automaticamente. Mutations nao sao retentadas por padrao.
retry: 1
3
Toast Notifications — Sonner
Erros de mutations e acoes do usuario sao exibidos via toast (Sonner). Mensagens amigaveis derivadas da resposta da API (error.response.data.message).
toast.error
4
Error Boundaries — Component Crash
React Error Boundaries capturam erros de renderizacao que quebrariam a pagina inteira. Exibem fallback UI ao inves de tela branca.
fallback UI
api-client.ts — Interceptor de Resposta
// Response interceptor - handle errors apiClient.interceptors.response.use( (response) => response, (error) => { // Handle 401 - unauthorized → auto logout if (error.response?.status === 401) { localStorage.removeItem('token') if (window.location.pathname !== '/login' && window.location.pathname !== '/register') { window.location.href = '/login' } } return Promise.reject(error) } )

10 Graceful Shutdown

O servidor escuta SIGINT e SIGTERM e faz shutdown ordenado em 4 fases, com timeout de drain de 30 segundos.

1
Stop Accepting Work
projectPathLock.cleanup() — libera locks de projeto.
stopMetricsAggregation() — para coleta de metricas.
Imediato
2
Drain Active Executions
Loop que verifica taskOrchestrator.getActiveCount() a cada 2 segundos.
Timeout maximo: 30 segundos (MAX_DRAIN_MS = 30_000).
Se exceder, forca shutdown com warning.
Max 30s
3
Shutdown Infrastructure
Desliga em ordem: stopTriggerScheduler() → stopExecutionWorker() → closeExecutionQueue() → app.close() → closeAllRedis()
Sequencial
4
Exit Process
process.exit(0) — encerra o processo com codigo de sucesso. Execucoes que ainda estavam rodando serao pegas pelo recovery no proximo startup.
exit(0)
server.ts — Graceful Shutdown (simplificado)
const signals = ['SIGINT', 'SIGTERM'] signals.forEach((signal) => { process.on(signal, async () => { console.log(`${signal} received, shutting down gracefully...`) // 1. Stop accepting new executions projectPathLock.cleanup() stopMetricsAggregation() // 2. Wait for active executions to drain (max 30s) const MAX_DRAIN_MS = 30_000 while (taskOrchestrator.getActiveCount() > 0 && Date.now() - drainStart < MAX_DRAIN_MS) { await new Promise(r => setTimeout(r, 2000)) } // 3. Shutdown infrastructure await stopTriggerScheduler() await stopExecutionWorker() await closeExecutionQueue() await app.close() await closeAllRedis() process.exit(0) }) })
