? O que sao Plugins

Plugins sao container packages que agrupam multiplos recursos em uma unica unidade instalavel. Um plugin pode conter MCP Servers, Skills, Agents e Rules — todos gerenciados como um conjunto coeso.

Instalacao atomica Toggle em cascata Delete em cascata Import de GitHub Resync de upstream
M
Manual

Criado via POST /plugins com um manifest JSON. O usuario define explicitamente quais MCP servers, skills e agents compoem o plugin.

source: "manual"

I
Importado

Importado de um repositorio GitHub via POST /plugins/import-url. O sistema escaneia o repo em busca de .mcp.json, SKILL.md e AGENT.md automaticamente.

source: "imported"

C Cascade Behavior

T
Toggle Cascade

PATCH /plugins/:id/toggle inverte o estado enabled do plugin e de todos os filhos atomicamente via $transaction.

🧩 Plugin.enabled = !current
→
🤖 McpServer.updateMany
→
🧠 Skill.updateMany
→
🎯 Agent.updateMany
const [plugin] = await prisma.$transaction([ prisma.plugin.update({ where: { id }, data: { enabled: next } }), prisma.mcpServer.updateMany({ where: { pluginId: id }, data: { enabled: next } }), prisma.skill.updateMany({ where: { pluginId: id }, data: { enabled: next } }), prisma.agent.updateMany({ where: { pluginId: id }, data: { enabled: next } }), ])
D
Delete Cascade

DELETE /plugins/:id remove o plugin e todos os recursos filhos atomicamente via $transaction com 4 operacoes.

McpServer.deleteMany
→
Skill.deleteMany
→
Agent.deleteMany
→
Plugin.deleteMany
await prisma.$transaction([ prisma.mcpServer.deleteMany({ where: { pluginId: id, userId } }), prisma.skill.deleteMany({ where: { pluginId: id, userId } }), prisma.agent.deleteMany({ where: { pluginId: id, userId } }), prisma.plugin.deleteMany({ where: { id, userId } }), ])

Nota sobre FK behavior: No schema Prisma, as FKs dos filhos usam onDelete: SetNull. Porem, o repository faz delete explicito via $transaction para garantir atomicidade e auditoria. Ownership e verificada em todas as operacoes: findFirst({ where: { id, userId } }).

I Installation Methods

1
Prepare o Manifest
Defina name, version, author e a lista de mcpServers, skills e agents no JSON.
2
POST /plugins
Envie o manifest no body. O sistema valida com Zod e verifica unicidade do nome.
3
Criacao Atomica
Plugin + todos os filhos sao criados em uma unica operacao Prisma nested create.
4
Audit Log
Registro de auditoria automatico via logAudit() com diff do estado final.
// POST /plugins { "name": "my-ai-toolkit", "description": "Toolkit com MCP servers e skills para IA", "version": "1.0.0", "author": "team", "manifest": { "mcpServers": [{ "name": "code-analyzer", "type": "stdio", "command": "npx", "args": ["@company/code-analyzer"] }], "skills": [{ "name": "code-review", "description": "Automated code review skill", "body": "Review the code for bugs and improvements...", "allowedTools": ["read_file", "search_code"] }], "agents": [{ "name": "reviewer", "systemPrompt": "You are a code reviewer...", "tools": ["read_file"], "model": "claude-sonnet-4-20250514" }] } }
1
Forneca a URL
URL de um repo GitHub. Pode incluir branch e subpath: github.com/user/repo/tree/main/plugins
2
Scan Automatico
O sistema usa a GitHub API para listar a arvore de arquivos e identifica .mcp.json, SKILL.md e AGENT.md/json.
3
Parse dos Recursos
Cada arquivo encontrado e baixado e parseado. Skills e agents usam frontmatter YAML.
4
Criacao do Plugin
Plugin criado com source: "imported", repoUrl salva para permitir resync futuro.
.mcp.json

Formato padrao Claude. Detecta mcpServers com command, args, env e type (stdio/http).

SKILL.md

Frontmatter YAML com name, description, allowed-tools, model. Body = skill content.

AGENT.md / .json

Agentes em MD (frontmatter + body) ou JSON (nome, systemPrompt, tools, permissionMode).

// POST /plugins/import-url { "url": "https://github.com/user/my-ai-tools", "projectPath": "/workspace/my-project" // opcional: salva skills localmente } // Resposta 201: { "id": "uuid...", "name": "my-ai-tools", "source": "imported", "createdAt": "..." }
R
Re-sync de Upstream

Para plugins importados do GitHub (source: "imported"), o endpoint POST /plugins/:id/resync re-busca a arvore do repositorio, re-baixa SKILL.md e AGENT.md/json atualizados, e opcionalmente reescreve os arquivos no projectPath local.

1
Valida Origem
Verifica se source == "imported" e se repoUrl existe. Rejeita plugins manuais.
2
Re-fetch Tree
Busca a arvore do repo via GitHub API, filtrando por SKILL.md e AGENT.md/json.
3
Download + Write
Baixa cada arquivo atualizado. Se projectPath definido, reescreve localmente com writeFileSafe().
4
Retorna Stats
Retorna { id, name, filesUpdated, lastSyncedAt } indicando quantos arquivos foram atualizados.
// POST /plugins/:id/resync { "projectPath": "/workspace/my-project" } // opcional override // Resposta 200: { "id": "uuid...", "name": "my-ai-tools", "filesUpdated": 3, "lastSyncedAt": "2026-03-24T15:30:00.000Z" }

R Relacao com Outros Recursos

Cada McpServer, Skill, Agent e Rule possui um campo opcional pluginId que aponta para o plugin pai. Quando o recurso nao pertence a nenhum plugin, pluginId e null.

Modelo Campo FK Tabela onDelete Index
McpServer pluginId String? mcp_servers SetNull @@index([pluginId])
Skill pluginId String? skills SetNull @@index([pluginId])
Agent pluginId String? agents SetNull @@index([pluginId])
Rule pluginId String? rules SetNull @@index (via table)
Recurso Standalone

Quando pluginId = null, o recurso foi criado independentemente. Gerenciado individualmente — nao e afetado por toggle/delete de plugins.

Recurso de Plugin

Quando pluginId != null, o recurso pertence a um plugin. Toggle e delete do plugin afetam esse recurso. O repository faz cascade explicito via $transaction.
