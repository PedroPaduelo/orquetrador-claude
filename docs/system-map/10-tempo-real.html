🔗 Fluxo de Conexao SSE

🎨

1. Frontend envia mensagem

POST /conversations/:id/messages/stream com { content, stepIndex?, attachments? }. AbortController criado para permitir cancelamento.

⚙

2. Backend valida e abre SSE

Verifica autenticacao, budget de tokens, projectPath. Seta headers SSE: Content-Type: text/event-stream, Cache-Control: no-cache, X-Accel-Buffering: no.

🔓

3. Registra listeners no EventEmitter

15 event handlers registrados em orchestratorEvents. Cada handler filtra por conversationId e escreve no raw response via sendEvent(name, data).

🤖

4. Orchestrator executa workflow

TaskOrchestrator chama o engine (Claude CLI). O engine emite eventos via orchestratorEvents.emit*() conforme progride pelos steps.

📡

5. Eventos fluem para o frontend

Cada evento e serializado como event: name\ndata: JSON\n\n. O frontend le via ReadableStream e processa no handleEvent().

✔

6. Conexao encerrada

Ao receber complete ou error, o backend faz cleanup dos listeners, libera lock do projectPath e chama reply.raw.end(). Frontend invalida queries do React Query.

💓 Heartbeat
A cada 15 segundos, o backend envia :heartbeat\n\n (SSE comment) para manter a conexao viva atraves de proxies e load balancers. O frontend nao precisa processar este evento.

🔄 Auto-Reconnect

Fluxo de Reconexao

1

Pagina carrega (mount)

useEffect no useSSEStream dispara checkAndReconnect()

2

Consulta status

GET /conversations/:id/execution-status retorna { active, state, pausedInfo }

3

Se state === 'running'

Chama watchExecution() que abre GET /conversations/:id/watch (SSE read-only)

4

Se state === 'paused'

Restaura setPaused(true, pausedInfo) para mostrar UI de input ao usuario

5

Se nao ativo

Watch endpoint retorna no_active_execution e fecha imediatamente. Nenhuma UI de streaming e mostrada.

Endpoints de Suporte

GET /conversations/:id/execution-status
Retorna estado atual da execucao. Checa primeiro in-memory (taskOrchestrator.isExecuting), depois paused state no DB.
{ active: boolean, state: 'running' | 'paused' | null, stepIndex: number | null, pausedInfo: object | null }
GET /conversations/:id/watch
SSE read-only: observa execucao em andamento sem iniciar nova. Mesmos 15 handlers que o stream principal. Se nenhuma execucao ativa, retorna no_active_execution e fecha.
POST /conversations/:id/cancel
Safety net para cancelamento. Frontend aborta via AbortController (que triggers backend close handler) e tambem chama este endpoint como fallback.

⚙ Configuracao
new Redis(REDIS_URL, { maxRetriesPerRequest: null, lazyConnect: true, connectionName: name, })
  • Lazy singletons — conecta apenas no primeiro acesso
  • maxRetriesPerRequest: null — necessario para BullMQ
  • connectionName — identifica pool em CLIENT LIST
💚 Graceful Degradation
Se Redis nao estiver disponivel, o sistema continua operando com funcionalidade reduzida:
  • Queue — getExecutionQueue() retorna null, execucao direta sem fila
  • Worker — startExecutionWorker() loga warning e nao cria worker
  • Cache — fallback para operacao sem cache (mais lento)
  • Rate limit — desabilitado quando Redis indisponivel

⏱ HTTP Timeout Configuration

Configuracoes criticas no server.ts para suportar conexoes SSE de longa duracao. Sem estas configuracoes, o Node.js encerraria conexoes SSE prematuramente.

keepAliveTimeout
10 min
headersTimeout
10 min + 1s
requestTimeout
0 (ilimitado)
socketTimeout
0 (ilimitado)
// server.ts - HTTP Server timeouts for SSE httpServer.keepAliveTimeout = 10 * 60 * 1000 // 10 min httpServer.headersTimeout = 10 * 60 * 1000 + 1000 // must be > keepAliveTimeout httpServer.requestTimeout = 0 // no limit for SSE streams
⚠ Por que headersTimeout > keepAliveTimeout?
O Node.js exige que headersTimeout seja maior que keepAliveTimeout. Se nao for, o servidor pode fechar conexoes keep-alive antes do timeout de headers, causando erros intermitentes em SSE.
