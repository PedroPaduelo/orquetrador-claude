Fluxo Principal de Execucao

O caminho completo de uma requisicao ate a resposta em streaming

Usuario
Frontend (React + Vite)
POST /stream (SSE Request)
Auth (Middleware JWT)
Orchestrator (Task Orchestrator)
Claude CLI (Execucao)
SSE Stream (Real-time)
Frontend (Renderiza)
Request Entry

Usuario envia mensagem via React frontend. O Axios faz POST para /conversations/:id/messages/stream. Headers SSE sao configurados imediatamente.

Processing

TaskOrchestrator coordena steps sequenciais ou em DAG paralelo. SessionManager mantem contexto. FileSync sincroniza arquivos por step. MemoryManager acumula contexto.

Output Delivery

Eventos SSE sao escritos no stream raw via reply.raw.writeHead(). Conteudo e actions chegam ao frontend em tempo real.

Ciclo de Vida dos Dados

As 6 fases que cada dado atravessa no sistema

FASE 1

Criacao

Dados criados pelo usuario atraves do formulario de mensagem ou upload de arquivos. Anexos incluem id, filename, mimeType, path, projectPath e URL.

content: string attachments[] stepIndex
FASE 2

Validacao

Schema Zod valida entrada. Auth middleware verifica JWT. TokenBudgetService verifica limites. validateProjectPath protege o filesystem.

Zod Schema JWT verify Budget check
FASE 3

Armazenamento

Prisma ORM persiste no PostgreSQL. Mensagens salvas via messagesRepository. Estados de execucao gerenciados pelo executionStateManager em memoria.

Prisma PostgreSQL Messages
FASE 4

Processamento

TaskOrchestrator executa via Claude CLI engine. Stream events parseados em tempo real. SessionManager + MemoryManager gerenciam contexto. runValidators valida output.

CliEngine SessionManager MemoryManager
FASE 5

Saida

Resultados via SSE em tempo real. webhooksService dispara notificacoes externas. API responses com JSON. FileSyncService sincroniza arquivos gerados de volta.

SSE Stream Webhooks FileSync
FASE 6

Arquivamento

Execucoes completadas mantidas no historico. Memory compactado para otimizar contexto futuro. ExecutionMonitor agrega metricas e logs de cada execucao.

Execution log Memory compact Monitor

Maquina de Estados — Step

Cada step individual dentro de uma execucao tem seu proprio ciclo de vida

pending
running
completed
failed
skipped

Transicao para skipped

Steps sao skipped quando um ConditionJump redireciona o fluxo para outro step, pulando intermediarios. Tambem em DAG quando dependencias nao sao satisfeitas.

Retry automático

Se StepConditions nao sao satisfeitas, o conditionsEvaluator pode disparar um condition:retry com contagem ate maxRetries.

Frontend ↔ Backend

POST /conversations/:id/messages/stream com JWT Bearer. Resposta: SSE stream com headers Content-Type: text/event-stream, X-Accel-Buffering: no.

Backend ↔ PostgreSQL

Prisma Client com queries tipadas. Conversation inclui Workflow + Steps (include com orderBy: stepOrder asc). messagesRepository salva user/assistant messages com metadata.

Backend ↔ Redis

sessionManager: get/set session tokens por conversationId+stepId. BullMQ: webhooks queue, cleanup jobs. @fastify/rate-limit: controle de requisicoes.

Backend ↔ Claude CLI

engine.execute(): child process com message, systemPrompt, resumeToken, model, projectPath, githubToken. onEvent: callback para stream events (content + action + metadata). onRawStdout/Stderr: raw output para ExecutionMonitor.

Fluxo de Eventos — OrchestratorEvents

16 tipos de eventos emitidos pelo sistema durante a execucao via EventEmitter

step:start

Step iniciado — id, nome, ordem, totalSteps

step:stream

Conteudo real-time: type content | action

step:complete

Step finalizado com content, sessionId, finished

step:error

Erro no step com stepName e mensagem

message:saved

Mensagem persistida: role, content, metadata

condition:retry

Retry N/maxRetries com retryMessage

condition:jump

Salto: fromStepId -> toStepId, toStepIndex

execution:complete

Execucao finalizada: success boolean

context:reset

Contexto resetado: stepName + reason

execution:cancelled

Cancelada: executionId + conversationId

dag:batch_start

Lote DAG: stepIds[] + batchIndex

validation:failed

Falha: validatorType + feedback

execution:paused

Pausa: resumeToken + askUserQuestion

execution:resumed

Retomada: stepName + stepOrder

user:interrupt

Interrupcao: stepName + userMessage

Sequencia Tipica de Eventos

step:start step:stream x N message:saved step:complete step:start ... execution:complete

Caminho do Evento ate o Frontend

1

TaskOrchestrator chama orchestratorEvents.emitStepStream() com executionId, stepId, type (content | action)

2

OrchestratorEvents (extends EventEmitter) emite 'step:stream'

3

execution.routes escuta o evento e escreve via reply.raw.write(`data: ${JSON.stringify(...)}\n\n`)

4

Frontend SSE Client recebe o evento e atualiza o estado React em tempo real

// Tipos de conteudo no stream

executionId: string

stepId: string
type: 'content' | 'action'

action?: { // tool use, system
type: string
name?: string
input?: unknown
output?: unknown
}
}
