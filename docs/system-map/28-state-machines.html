Trace Result Status

Cada trace de execucao (ExecutionTrace) possui um resultStatus que indica o resultado daquele step individual. Nao e uma maquina de estado com transicoes, mas um enum de resultados finais.

✓ success
✗ error
⏱ timeout
⛔ cancelled
✍ needs_input
⚠ interrupted
StatusClassificacaoDescricaoProximo passo?
success Terminal Step executou com sucesso. Output disponivel no trace. Avanca para proximo step do workflow.
error Terminal Erro durante execucao. classifyError() define errorCategory. Retry se retryCount < maxRetries, senao falha.
timeout Terminal Step excedeu o tempo maximo (timeoutMs). Retry ou falha dependendo da configuracao.
cancelled Terminal Execucao cancelada pelo usuario via API. Nenhum. Execucao inteira e marcada como cancelled.
needs_input Pausavel Claude pede input do usuario (stopReason: needs_input). Execucao pausa. Retoma quando usuario fornece input.
interrupted Terminal Processo interrompido (SIGTERM, crash, server restart). Recovery tenta retomar do ultimo step salvo.

State Persistence

O estado de cada execucao e persistido no PostgreSQL via model ExecutionState. Isso permite recovery completo apos restart do servidor.

ExecutionState Model

id String @id UUID da execucao
status ExecutionStatus Estado atual da maquina de estado
currentStepIndex Int Indice do step sendo executado
retryCounts Json Map stepId → retryCount
metadata Json Contexto adicional (error info, lastOutput)
startedAt DateTime Timestamp de inicio
completedAt DateTime? Timestamp de finalizacao (null se em andamento)

Recovery on Server Restart

Step 1 Detecta execucoes interrompidas

Query: status = 'running' no banco. Essas nao terminaram normalmente.

Step 2 Merge metadata

Combina metadata existente com info de recovery (recoverAt, previousStep).

Step 3 Resume do currentStepIndex

Re-enfileira no BullMQ a partir do step onde parou. Nao re-executa steps ja completos.

Step 4 Graceful Shutdown

30s drain period. SIGTERM salva estado atual antes de encerrar.

// Recovery simplificado (server startup) const stuckExecutions = await prisma.executionState.findMany({ where: { status: 'running' } }); for (const exec of stuckExecutions) { await prisma.executionState.update({ where: { id: exec.id }, data: { metadata: { ...exec.metadata, recoveredAt: new Date(), previousStep: exec.currentStepIndex } } }); // Re-enqueue from currentStepIndex await executionQueue.add('resume', { executionId: exec.id, fromStep: exec.currentStepIndex }); }
