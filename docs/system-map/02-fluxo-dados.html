Fluxo de Dados

Ciclos de vida das entidades, fluxo end-to-end de execucao, eventos em tempo real, controle de tokens e trigger scheduler.

📋 1. Ciclo de Vida do Workflow

Um Workflow representa uma sequencia (ou DAG) de steps a serem executados. Tem tres estados possiveis, controlados pelo enum WorkflowStatus.
📝 Draft
→
✅ Active
→
📦 Archived
← pode voltar de Archived para Active
Estado Descricao Acoes Permitidas
draft Workflow em construcao. Steps podem ser adicionados/removidos/reordenados. Editar steps, editar config, ativar (→ active)
active Workflow pronto para execucao. Conversations podem ser criadas a partir dele. Executar, criar conversation, editar steps, arquivar (→ archived)
archived Workflow descontinuado. Nao aceita novas execucoes, mas o historico e mantido. Visualizar, reativar (→ active), excluir permanentemente

❌ Failed

Fork de Conversations

Qualquer conversation pode ser "forkada" criando uma nova conversation com parentId apontando para a original. A relacao e mantida via @relation("ConversationParent"). O fork herda o historico de mensagens ate o ponto do fork, mas segue um caminho independente a partir dai.
Conversation A (parent)
──fork──→
Conversation B (parentId: A)
Estado Descricao Transicao
active Conversation em andamento. Aceita novas mensagens e execucoes. → completed, → failed, → archived
completed Conversation finalizada com sucesso. Apenas leitura. Terminal (pode ser forked)
failed Conversation falhou. Erro critico no workflow ou na engine. Terminal (pode ser forked)
archived Conversation desativada manualmente pelo usuario. Terminal

⚙️ 3. Ciclo de Vida da Execution

Cada mensagem enviada em uma conversation gera um ExecutionState. Controlado pelo enum ExecutionStateStatus com 6 estados possiveis.
📥 Queued
→
🔄 Running
↕ pause/resume
⏸ Paused
↓ resultado final
✅ Completed
todos steps ok
❌ Failed
erro nao-recuperavel
🚫 Cancelled
cancelado pelo usuario

Transicoes e Triggers

De Para Trigger
queued running Worker pega o job da fila BullMQ e inicia o TaskOrchestrator
running paused Usuario envia PUT /executions/:id/pause ou step retorna needsUserInput: true
paused running Usuario envia PUT /executions/:id/resume com nova mensagem
running completed Todos os steps terminaram com sucesso (ou skip)
running failed Step falha e errorHandling = "fail" (ou max retries atingido)
running cancelled Usuario envia DELETE /executions/:id ou timeout global
paused cancelled Usuario cancela enquanto execucao esta pausada

🔀 4. Fluxo Completo de Execucao (End-to-End)

Desde o momento em que o usuario envia uma mensagem ate a conclusao da execucao, todos os componentes do sistema participam em sequencia.

A Entrada da Mensagem

1
Usuario envia mensagem
Frontend faz request para o backend via SSE.
POST /api/conversations/:id/messages/stream
2
Mensagem salva no banco
A mensagem do usuario e persistida como Message com role: 'user' associada a conversation e ao step atual.
3
ExecutionState atualizado para 'running'
O executionStateManager cria ou atualiza o registro com state: 'running', incrementa currentStepIndex.

B Loop de Execucao dos Steps

Para cada WorkflowStep (sequencial ou via DAG), o TaskOrchestrator executa o seguinte pipeline:
4
Avaliar skip conditions
O metodo evaluateSkipCondition() verifica se o step deve ser pulado. Condicoes suportadas:
- empty_input — pula se a mensagem estiver vazia
- contains:keyword — pula se contem a keyword
- not_contains:keyword — pula se NAO contem
- step_index > N — pula baseado no indice
- true / false — literal
5
Construir contexto
Monta o contexto de execucao combinando:
- System prompt: prompt base + step.systemPrompt + memoria salva
- Resume token: recuperado do SessionManager para continuar sessao do Claude
- Attachments: arquivos enviados pelo usuario
- Git config: configura user.name/email e GitHub token no workspace
6
Compactar mensagens antigas (Memory Manager)
Se o contexto ficar muito longo ("prompt_too_long"), o MemoryManager e acionado: resume a conversa, salva o sumario como memoria, limpa a sessao e recria com contexto compacto.
memoryManager.summarizeAndSave(conversationId, stepId)
7
Executar ClaudeCodeEngine
Chama engine.execute() que spawna o processo claude CLI com sudo -u claude-sandbox. A engine roda em isolamento, recebendo o prompt e retornando streaming JSON events.
ClaudeCodeEngine.execute({ message, systemPrompt, resumeToken, ... })
8
Parsear streaming output
O StreamParser recebe chunks JSON do stdout do processo e emite eventos tipados: content, action, metadata. O ExecutionMonitor coleta metricas em tempo real.
9
Salvar ExecutionTrace
O monitor faz flush() ao final do step, criando um ExecutionTrace com: timing (startedAt, completedAt, durationMs), tokens (input/output), stdout acumulado, stderr, status, exitCode, signal.
10
Extrair ExecutionToolCalls
O ToolCallExtractor analisa o conteudo das actions e extrai todas as chamadas de ferramentas (Read, Write, Bash, etc.) com seus argumentos e resultados. Salvas como ExecutionToolCall.
11
Salvar Message (assistant)
O conteudo gerado pelo Claude e salvo como Message com role: 'assistant', incluindo metadata com actions, sessionId (resume token) e stepName.
12
Executar validators
Se o step tem ValidatorConfig, o runValidators() executa as validacoes no output. Se falhar, emite validation_failed via SSE.
13
Avaliar conditions (retry, jump, complete)
O ConditionsEvaluator analisa o StepConditions do step:
- retry: re-executa o step (com backoff exponencial, ate maxRetries)
- jump: pula para outro step pelo nome/indice
- complete: finaliza a execucao antecipadamente
14
Tratar erros (errorHandling)
O StepErrorHandler decide o que fazer baseado em step.errorHandling:
- fail: para tudo, execution → failed
- skip: pula o step e continua com o proximo
- continue_next: registra erro e continua para o proximo step
- fallback: executa com FALLBACK_MESSAGE definido
15
Emitir SSE events para frontend
A cada etapa, o orchestratorEvents emite eventos via EventEmitter, que sao convertidos em SSE e enviados ao frontend em tempo real.

C Finalizacao

16
Criar ExecutionAggregate
createOrUpdateAggregate() consolida todos os traces da execucao em um unico registro ExecutionAggregate com totais de tokens, custo, duracao e status.
17
Atualizar DailyExecutionMetrics
Metricas diarias sao incrementadas: totalExecutions, successCount/failureCount, tokens, custo, duracoes (avg, p50, p95). Atualizadas atomicamente.
18
Dispatch webhooks
O webhooksService verifica se existem webhooks configurados para o evento e dispara para cada URL registrada. Cria WebhookDelivery com status (pending → success/failed).
19
ExecutionState → completed/failed
O estado final e atualizado. O evento execution:complete (ou execution:cancelled) e emitido via SSE, sinalizando ao frontend que a execucao terminou.

⏰ 7. Fluxo de Dados do Trigger Scheduler

O Trigger Scheduler permite agendar execucoes automaticas de workflows com base em expressoes cron. O worker roda a cada 30 segundos verificando execucoes pendentes.

Ciclo de Vida da ScheduledExecution

📋 Pending
→
🔄 Running
↓
✅ Completed
❌ Failed
🚫 Cancelled

Fluxo End-to-End do Trigger Cron

1. WorkflowTrigger Criado
Admin cria um trigger com type: 'cron', cronExpr: '0 9 * * *' (ex: todo dia as 9h) e associa a um workflow. Armazena timezone opcional.
2. scheduleCron()
O metodo triggersService.scheduleCron(triggerId) calcula a proxima execucao via parseNextCron() e cria um registro ScheduledExecution com status: 'pending' e scheduledAt no futuro.
3. Worker Poll (cada 30s)
O scheduler worker roda periodicamente (setInterval 30s), chamando triggersService.processScheduledExecutions(). Busca todos os ScheduledExecution com status: 'pending' e scheduledAt <= now.
4. Status → running
Para cada execucao devida, atualiza status: 'running' e executedAt: now. Tambem atualiza o lastTriggeredAt do WorkflowTrigger.
5. Criar Conversation + Enqueue
Cria uma nova Conversation para o workflow associado e enfileira a execucao na fila BullMQ (execution-queue). O job contem conversationId, workflowId e metadata do trigger.
6. ExecutionWorker Processa
O ExecutionWorker pega o job da fila e executa via TaskOrchestrator — o mesmo fluxo de execucao descrito na secao 4 acima.
7. Status → completed/failed
Ao final, o ScheduledExecution e atualizado com status final (completed ou failed) e o campo result contem o JSON do resultado ou errorMessage.
8. Re-agendar (se recorrente)
Se o trigger e recorrente (cron), scheduleCron() e chamado novamente para criar a proxima ScheduledExecution baseada na expressao cron.

Modelo WorkflowTrigger

Campo Tipo Descricao
typeStringTipo do trigger: "cron", "webhook", "manual"
cronExprString?Expressao cron de 5 campos (ex: "0 9 * * *")
cronTimezoneString?Timezone para calculo (ex: "America/Sao_Paulo")

Acoes Disponiveis

▶️

Executa o trigger imediatamente, independente do cron. Cria ScheduledExecution com scheduledAt = now.
⏹️

Cancela todas as ScheduledExecutions pendentes de um trigger. Marca como cancelled.
