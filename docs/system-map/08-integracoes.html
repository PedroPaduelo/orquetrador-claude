Claude CLI Integration
Motor de execucao via subprocess — JSON stream, resume tokens, sandbox
Core
▼

Fluxo de Dados

📋 TaskStep
→
⚙️ ClaudeCodeEngine
→
🚀 spawn(claude)
→
📡 JSON Stream
→
📝 Parse Output
→
💾 Save Result
Processo
ClaudeCodeEngine faz spawn() de um subprocess claude CLI com flags --json e --model. Roda como usuario claude-sandbox via sudo -u para isolamento.
JSON Stream Output
Cada linha do stdout e um objeto JSON. Tipos: assistant (texto), tool_use (chamada de ferramenta), tool_result, system. Parseado em real-time e enviado via SSE ao frontend.
Resume Tokens
Apos cada execucao, o CLI retorna um resume_token. Armazenado no banco no campo Execution.resumeSessionId. Usado para manter contexto entre steps sem reenviar historico completo.
Selecao de Modelo
Configuravel por step via --model flag. Modelos disponiveis: claude-sonnet-4-20250514, claude-opus-4-0-20250514, claude-haiku. Default do projeto ou override por execucao.
Tool Calls
Extraidos do stream JSON como eventos tool_use. Cada tool call registrado em StepToolCall com nome, input, output e duracao. Permite auditoria completa.
Sandbox Isolation
Controlado por env SANDBOX_ENABLED. Quando ativo, execucoes rodam como claude-sandbox (sem login shell, permissoes restritas). Script setup-sandbox.sh configura usuario e sudoers.

Exemplo de Spawn

// ClaudeCodeEngine.execute() const args = [ '--json', '--model', step.model || 'claude-sonnet-4-20250514', '--max-turns', 25, ]; if (resumeToken) args.push('--resume', resumeToken); const proc = spawn('sudo', ['-u', 'claude-sandbox', 'claude', ...args]); proc.stdout.on('data', (chunk) => { const lines = chunk.toString().split('\n'); for (const line of lines) { const event = JSON.parse(line); // Envia via SSE + salva no banco handleStreamEvent(event); } });
💡 Arquitetura

O engine nao usa a API REST do Claude diretamente — ele opera via CLI binary, que gerencia contexto, rate limits e tool permissions internamente. Isso permite usar features exclusivas do CLI como resume, MCP tool passthrough e file system access.

Multi-conta
Cada usuario pode vincular multiplas contas GitHub via GitAccount. Cada conta tem um label descritivo (ex: "Pessoal", "Trabalho") e token OAuth encriptado.
Token Storage
Token OAuth armazenado em GitAccount.accessToken com encriptacao AES-256. Scopes necessarios: repo, read:org. Refresh automatico quando suportado.
Import Types
Importa 3 tipos de recurso de repos: Skills (prompts/templates), Agents (configuracoes de agente), Rules (regras e hooks).
File Manifest
Cada recurso importado tem um fileManifest JSON que rastreia: path, SHA, tamanho e ultima sincronizacao de cada arquivo. Permite diff incremental.
Sync Modes
Manual: usuario clica "Sync" na UI.
Auto: via triggers/webhooks do GitHub (push events). Compara manifests e importa apenas o que mudou.
Branch Selection
O usuario seleciona a branch de onde importar. Default: branch principal do repo. Listada via endpoint /branches.

Endpoints da API

GET /api/git-accounts/:id/repos Lista repositorios da conta GitHub
GET /api/git-accounts/:id/repos/:owner/:name/branches Lista branches do repositorio
GET /api/git-accounts/:id/repos/:owner/:name/files Lista arvore de arquivos (branch + path)
POST /api/git-accounts Registra nova conta Git (label + token)
POST /api/skills/import-github Importa skill de um repo + branch
POST /api/agents/import-github Importa agent de um repo + branch

Tool Discovery
Ao conectar, o Orquetrador chama tools/list no MCP server. Retorna nome, descricao, inputSchema (JSON Schema) de cada tool disponivel. Armazenado em McpServer.toolsCache.
Connection Testing
Endpoint POST /api/mcp-servers/:id/test verifica conectividade, faz handshake e valida que tools sao listadas. Retorna status + latencia.
Plugin Association
MCP servers sao vinculados a Plugins. Quando um plugin e ativado em uma execucao, suas tools MCP sao injetadas no system prompt do Claude.
Passthrough ao Claude
Tools MCP sao formatadas como allowed_tools no prompt. O Claude chama a tool → Orquetrador roteia para o MCP server → resposta volta ao Claude.

Redis Integration
3 connection pools — rate limiting, JWT cache, BullMQ, Pub/Sub com graceful degradation
Cache
▼

3 Connection Pools

General

Pool principal para operacoes de leitura/escrita: rate limiting counters, JWT blocklist, cache geral, session data.

GET/SET/INCR
Subscriber

Pool dedicada a subscricoes Pub/Sub. Recebe eventos de execucao (progress, completion) e distribui para conexoes SSE ativas.

SUBSCRIBE
Publisher

Pool dedicada a publicacao de eventos. O engine publica progress de execucao, o backend publica notificacoes.

PUBLISH

Casos de Uso

Rate Limiting
Contadores por IP e por usuario com TTL. Keys: rl:{ip}:{window}. Usa INCR + EXPIRE atomicos. Configuravel por rota via @fastify/rate-limit.
JWT Cache / Blocklist
Tokens revogados armazenados em jwt:blocked:{jti} com TTL igual ao tempo restante do token. Verificacao em cada request autenticado.
BullMQ Queue
Fila de jobs para execucoes assincronas. Queue name: execution-queue. Usa Redis como broker. Workers consomem e processam via ClaudeCodeEngine.
Pub/Sub
Canais: execution:{id}:progress, execution:{id}:complete. Frontend conecta via SSE, backend subscribe no canal Redis e faz bridge.
⚠️ Graceful Degradation

O sistema funciona sem Redis. Quando a conexao falha: rate limiting usa fallback in-memory, BullMQ nao processa jobs (execucoes ficam pendentes), Pub/Sub desabilita streaming em tempo real (frontend faz polling). O backend loga warning mas nao crash. Redis e optional para dev local.

Eventos Suportados

execution_started execution_completed execution_failed step_started step_completed step_failed task_created task_completed
HMAC-SHA256 Signing
Cada webhook tem um secret. O payload e assinado com HMAC-SHA256 e enviado no header X-Webhook-Signature. O receptor valida: hmac(secret, body) === signature.
Delivery Tracking
Cada envio gera um WebhookDelivery com: status HTTP, response body, latencia, timestamp. Historico completo para debug.
Retry Policy
Se o endpoint retorna status >= 400 ou timeout, o sistema faz retry com backoff exponencial. Maximo de 3 tentativas com delays de 10s, 60s, 300s.
Test Endpoint
POST /api/webhooks/:id/test envia um evento de teste para o URL configurado. Valida conectividade e signature sem gerar dados reais.

Payload de Exemplo

// POST https://seu-server.com/webhook // Headers: // Content-Type: application/json // X-Webhook-Signature: sha256=a1b2c3d4... // X-Webhook-Event: execution_completed // X-Webhook-Id: wh_abc123 { "event": "execution_completed", "timestamp": "2026-03-24T10:30:00Z", "data": { "executionId": "exec_xyz789", "taskId": "task_abc123", "status": "completed", "duration": 45200, "stepsCompleted": 5, "tokensUsed": 12450 } }

Autenticacao JWT

Token Flow
Login → backend gera JWT (access + refresh) → frontend armazena em memory (Zustand store) → Axios interceptor injeta Authorization: Bearer {token} em cada request.
Token Refresh
Quando access token expira (401), interceptor Axios faz request automatico de refresh. Se refresh falha, redireciona para login. Sem localStorage para tokens (security).
React Query Cache
Todas as queries usam TanStack React Query. Cache com staleTime configuravel por recurso. Invalidacao automatica apos mutations (create/update/delete).
Error Handling
Interceptor global trata erros: 401 → refresh/redirect, 403 → toast "sem permissao", 500 → toast generico. React Query retry: 3x com backoff para network errors.

🔒 CORS & Security

Backend configura CORS via @fastify/cors com origin whitelist. Apenas o dominio do frontend (orquestrador.ddw1sl.easypanel.host) e permitido. Credentials: true para cookies de refresh token. CSP headers configurados para prevenir XSS.
