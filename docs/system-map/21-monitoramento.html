📊 4 Camadas de Observabilidade

O sistema de monitoramento opera em 4 granularidades distintas. Cada camada captura dados em uma frequencia diferente e serve a casos de uso especificos, desde debug em tempo real ate analytics de longo prazo.

Real-time segundo a segundo
⏱ ExecutionTrace
🔧 ExecutionToolCall
Cada step de execucao gera um trace com timing, tokens, custo e tool calls extraidas
Logging eventos estruturados
📝 ExecutionLog
12 event types: step_start, step_complete, step_fail, workflow_start, retry, cancel, etc.
Hourly agregacao horaria
💰 HourlyTokenUsage
Rollup horario de tokens e custo por modelo e usuario
Daily resumo diario
📅 DailyExecutionMetrics
📊 ExecutionAggregate
⭐ ResourcePerformanceMetric
Metricas pre-agregadas para dashboards e analytics rapidos

⏱ ExecutionTrace — Anatomia do Trace

Cada step de execucao gera um ExecutionTrace detalhado. Ele captura desde o comando executado ate o resultado final, com timeline de eventos parseados.

💻
Command & Environment
commandLine envSnapshot systemPrompt model projectPath messageLength
📤
Output & Events
stdoutRaw stderrRaw parsedEvents[]

parsedEvents contem a timeline completa: cada evento JSON emitido pelo Claude CLI e parseado em sequencia

⏰
Timing
startedAt firstByteAt firstContentAt completedAt durationMs

4 timestamps permitem medir latencia de conexao (startedAt → firstByteAt), time-to-first-content, e duracao total

✅
Result
exitCode signal resultStatus errorMessage errorCategory errorCode
💰
Tokens & Cost
inputTokens outputTokens cacheCreationInputTokens cacheReadInputTokens totalCostUsd
⚡
Performance
durationApiMs numTurns stopReason

🔧 ExecutionToolCall

Tool calls sao extraidas de cada trace pelo ToolCallExtractor. Cada chamada de ferramenta feita pelo Claude e registrada individualmente com input, output, duracao e status.

Campo Tipo Descricao
toolName String Nome da ferramenta (ex: Read, Write, Bash, Grep)
input JSON Parametros enviados para a ferramenta
output JSON Resultado retornado pela ferramenta
durationMs Int Tempo de execucao em milissegundos
success Boolean Se a chamada foi bem-sucedida
errorMessage String? Mensagem de erro (se falhou)

📝 ExecutionLog — Audit Trail

Cada evento significativo durante uma execucao e registrado como um ExecutionLog. Funciona como audit trail completo do ciclo de vida de workflows e steps.

Lifecycle
workflow_start workflow_complete workflow_fail
Steps
step_start step_complete step_fail step_error step_skip
Control
retry pause resume cancel
// ExecutionLog schema { "id": String // UUID "executionId": String // FK para Execution "stepId": String? // FK para ExecutionStep (opcional) "eventType": String // step_start | step_complete | ... "data": Json // Contexto do evento (free-form) "createdAt": DateTime // Timestamp do evento }

O campo data e JSON livre e contem contexto especifico do evento. Ex: para step_fail, contem errorMessage e errorCategory. Para retry, contem attempt number e backoff delay.

📊 ExecutionAggregate

Resumo cross-step de uma execucao completa. Gerado pelo MetricsAggregator apos a finalizacao do workflow.

🔢
Contadores
totalSteps completedSteps failedSteps skippedSteps

Contagem por status de cada step. totalSteps = completedSteps + failedSteps + skippedSteps

💰
Tokens & Custo
totalInputTokens totalOutputTokens totalCostUsd

Soma de todos os traces. Custo calculado com pricing table por modelo.

🎯
Final Status

O campo finalStatus reflete o resultado geral da execucao:

completed failed partial cancelled

partial = pelo menos 1 step completou mas houve falhas. cancelled = interrompido pelo usuario ou timeout.

📅 DailyExecutionMetrics

Metricas pre-agregadas por dia. Geradas pelo MetricsAggregator e consumidas pelos dashboards sem necessidade de queries pesadas.

Campo Tipo Descricao
date DateTime Data de referencia (dia)
totalExecutions Int Total de execucoes no dia
successCount Int Execucoes com sucesso
failureCount Int Execucoes com falha
avgDurationMs Float Duracao media das execucoes
p50DurationMs Float Percentil 50 (mediana) de duracao
p95DurationMs Float Percentil 95 de duracao (tail latency)
errorBreakdown JSON Contagem por errorCategory { "timeout": 3, "config_error": 1 }
modelBreakdown JSON Contagem por modelo usado { "opus-4": 12, "sonnet-4": 8 }

🎯 ResourcePerformanceMetric

Tracking de performance por recurso individual (skill, agent ou rule). Permite identificar quais recursos sao mais eficientes, mais caros ou mais propensos a falhas.

📊
Campos
resourceId resourceType executionCount successCount avgDurationMs p95DurationMs totalCostUsd avgQualityScore
💡
Uso
  • • Identificar skills lentas (p95 alto)
  • • Detectar agents com baixa taxa de sucesso
  • • Comparar custo entre recursos similares
  • • Ranquear por qualidade (avgQualityScore)

⚙ Monitoring Services

5 servicos backend que coletam, processam e agregam dados de monitoramento.

👁
ExecutionMonitor
Wraps engine execution. Registra startedAt, firstByteAt, firstContentAt e completedAt. Calcula durationMs automaticamente. Ponto de entrada principal para traces.
real-time per-step
📊
MetricsAggregator
Fire-and-forget pos-execucao. Gera ExecutionAggregate e atualiza DailyExecutionMetrics. Nao bloqueia o fluxo principal.
async fire-and-forget
🔧
ToolCallExtractor
Parseia parsedEvents[] dos traces e extrai cada tool_use/tool_result como ExecutionToolCall individual. Calcula duracao por ferramenta.
parser per-trace
💰
HourlyTokenAggregator
Roda a cada hora via BullMQ repeatable job. Agrupa tokens e custo por modelo e usuario. Alimenta HourlyTokenUsage.
cron hourly
🧹
TraceCleanup
Delecao periodica de traces antigos. Configurable retention period. Preserva aggregates mesmo apos deletar traces brutos.
cron cleanup
