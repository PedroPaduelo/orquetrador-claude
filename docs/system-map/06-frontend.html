1 Arquitetura em Camadas

Organização feature-first com camadas claras de responsabilidade

ROUTING LAYER app/routes.tsx

React Router v7 com 20 rotas lazy-loaded. ProtectedRoute via JWT. Retry automático em chunk 404 (stale deploys).

LAYOUT LAYER app/app-layout.tsx + sidebar.tsx

AppLayout 2-pane com Sidebar colapsável (w-60). Sidebar: 3 seções (Menu 5 itens, Config 9 itens, Admin condicional). Dark mode nativo.

FEATURE LAYER features/{domain}/

16 domínios independentes. Cada feature: index.tsx (página), api.ts (HTTP), store.ts (Zustand), types.ts, hooks/, components/.

STATE LAYER Zustand + React Query v5

Zustand para estado local/streaming (5 stores). React Query para server cache (staleTime 30s, gcTime 5min, retry 1). SSE customizado para real-time.

API LAYER shared/lib/api-client.ts + sse-client.ts

Axios com interceptors (auto-inject JWT, 401→logout). SSE client custom via fetch (não EventSource). Proxy /api → :3333.

Auto-Reconnect

1.Ao montar, checa /execution-status
2.Se running → conecta /watch (SSE)
3.Se paused → restaura pausedInfo + AskUserQuestion
4.Processa eventos identicamente ao sendMessage

Cancel Flow

1.Flag cancelledRef = true
2.AbortController.abort() (HTTP fecha)
3.Steps running → status cancelled
4.POST /cancel (safety net)
5.invalidateQueries para estado final

6 Workflow Wizard

Criacao e edicao de workflows em 3 fases com 12 sub-componentes

1

Basic Info

Nome (obrigatorio), descricao, tipo: sequential (auto) ou step_by_step (usuario controla).

2

Step Builder

Split panel: sidebar com lista de steps + editor com systemPrompt, resources (MCP, Skills, Agents, Rules, Hooks), validators, variables, DAG deps.

3

Review

Resumo completo: workflow info, flow preview, cards de cada step com resource counts e configuracoes. Botao "Criar" ou "Salvar".

WorkflowStep Interface

interface WorkflowStep {
  name: string              // obrigatorio
  systemPrompt?: string       // prompt customizado
  useBasePrompt?: boolean     // combina com base prompt
  conditions: StepConditions   // routing rules
  maxRetries: number         // retry on failure
  mcpServerIds: string[]     // MCP servers
  skillIds: string[]         // skills
  agentIds: string[]         // agentes
  ruleIds: string[]          // rules
  hookIds: string[]          // hooks
  dependsOn?: string[]       // DAG dependencies
  validators?: ValidatorConfig[] // build_check, test_check, llm_judge
  outputVariables?: string[] // variaveis produzidas
  inputVariables?: string[]  // variaveis necessarias
  errorHandler?: 'fail' | 'skip' | 'fallback' | 'continue_next'
}
