📦 Visao Geral

O Orquetrador Claude usa BullMQ 5 com Redis (via ioredis) para processar execucoes de workflows em background. Toda execucao passa pela fila antes de chegar ao TaskOrchestrator.

A arquitetura desacopla a API (que recebe o request) do processamento real (que roda no worker), permitindo concorrencia controlada e resiliencia a falhas.

⚙️ Opcoes de Job (defaultJobOptions)

🗑️
removeOnComplete
{ count: 100 }

Mantem ate 100 jobs completados no Redis. Jobs mais antigos sao removidos automaticamente para evitar crescimento indefinido da memoria.

❌
removeOnFail
{ count: 50 }

Mantem ate 50 jobs falhados para analise. Permite diagnostico de erros sem acumular dados indefinidamente.

🔄
attempts
1

Sem auto-retry pelo BullMQ. O TaskOrchestrator tem sua propria logica de retry com backoff e classificacao de erros.

new Queue<ExecutionJobData>('execution', { connection: getRedis() as any, defaultJobOptions: { removeOnComplete: { count: 100 }, removeOnFail: { count: 50 }, attempts: 1, // Execution should not auto-retry }, })

🔧 Configuracao do Worker

⚡
Concorrencia

Controlada pela env EXECUTION_CONCURRENCY (default: 3).

Determina quantos jobs podem ser processados simultaneamente pelo worker. Cada job consome um processo Claude, entao o valor deve ser ajustado conforme recursos disponiveis.

🛡️
Inicializacao Resiliente

O worker inicia junto com o servidor mas de forma nao-bloqueante.

Se o Redis nao estiver disponivel, a fila e desabilitada silenciosamente (warn no log) e o servidor continua funcionando sem processamento de jobs.

Eventos do Worker

Evento Quando Acao
completed Job processado com sucesso Log: Job {id} completed for conversation {conversationId}
failed Job falhou durante processamento Log error: Job {id} failed: {error.message}
error Erro interno do worker (conexao, etc) Log error: [ExecutionWorker] Error: {message}
const CONCURRENCY = parseInt(process.env.EXECUTION_CONCURRENCY || '3', 10) const _worker = new Worker<ExecutionJobData>( 'execution', processExecutionJob, { connection: getRedis() as any, concurrency: CONCURRENCY, } ) _worker.on('completed', (job) => { /* log success */ }) _worker.on('failed', (job, err) => { /* log error */ }) _worker.on('error', (err) => { /* log error */ })

Active → Completed processExecutionJob() retorna
Active → Failed processExecutionJob() lanca erro

Detalhes do Processamento

Active: O processor carrega a conversation + workflow do banco, monta o contexto e decide o modo de execucao:

— sequential + DAG → executeDAG()
— sequential → executeSequential()
— step_by_step → executeStepByStep()

Erros: Publicados via Redis Pub/Sub no canal execution:{conversationId} para o frontend receber em tempo real.

🔴 Conexao Redis

O Redis e configurado via REDIS_URL (default: redis://localhost:6379). Todas as conexoes usam lazy connect — so conectam quando acessadas pela primeira vez.

Se o Redis nao estiver disponivel, o sistema degrada gracefully: a fila e desabilitada, o worker nao inicia, mas a API continua respondendo (execucoes sincronas diretas, sem fila).

🛑 Graceful Shutdown

O servidor escuta SIGINT e SIGTERM e executa um shutdown ordenado para garantir que nenhuma execucao seja perdida.
1. Parar de Aceitar Novos Jobs
projectPathLock.cleanup() — Libera locks de projeto.
stopMetricsAggregation() — Para o agregador de metricas.
2. Drenar Execucoes Ativas (max 30s)
Loop que verifica taskOrchestrator.getActiveCount() a cada 2s.
Se ainda houver execucoes apos 30s, forca o shutdown com warning no log.
3. Parar Trigger Scheduler
stopTriggerScheduler() — Para o scheduler de triggers cron/webhook.
4. Parar Worker
stopExecutionWorker() — Worker para de pegar novos jobs e fecha a conexao.
5. Fechar Queue
closeExecutionQueue() — Fecha a instancia da fila BullMQ.
6. Fechar Fastify
app.close() — Fecha o servidor HTTP/WebSocket.
7. Fechar Redis
closeAllRedis() — Encerra as 3 conexoes Redis (general, subscriber, publisher) via Promise.allSettled().
const signals = ['SIGINT', 'SIGTERM'] signals.forEach((signal) => { process.on(signal, async () => { // 1. Stop accepting new executions projectPathLock.cleanup() stopMetricsAggregation() // 2. Wait for active executions to drain (max 30s) const MAX_DRAIN_MS = 30_000 while (activeCount > 0 && elapsed < MAX_DRAIN_MS) { /* poll */ } // 3-7. Shutdown infrastructure await stopTriggerScheduler() await stopExecutionWorker() await closeExecutionQueue() await app.close() await closeAllRedis() process.exit(0) }) })

🔧 Recovery de Crashes

🔍
recoverStaleExecutions()

Executada no startup do servidor. Busca no banco todas as ExecutionState com state: 'running' — ou seja, execucoes que estavam ativas quando o servidor caiu.

Para cada execucao stale:

  • Marca como failed
  • Preserva metadata existente via merge
  • Adiciona recoveryError e recoveryFailedAt
export async function recoverStaleExecutions(): Promise<number> { const staleExecutions = await prisma.executionState.findMany({ where: { state: 'running' }, select: { id: true, metadata: true }, }) for (const exec of staleExecutions) { await prisma.executionState.update({ where: { id: exec.id }, data: { state: 'failed', metadata: { ...existingMeta, recoveryError: 'Server restarted during execution', recoveryFailedAt: new Date().toISOString(), }, }, }) } }

🎯 Funcao Processadora

⚡
processExecutionJob(job)
Funcao principal que o Worker executa para cada job da fila
Fluxo Completo do Processor ▼
1
Extrair dados do job

Desestrutura conversationId, content, stepIndex, projectPath, workflowType, userId, attachments do job.data

2
Obter canal Redis Pub/Sub

Canal: execution:{conversationId} — usado para enviar eventos ao frontend via SSE

3
Carregar Conversation + Workflow do banco

prisma.conversation.findUnique() com include de workflow e steps (ordenados por stepOrder)

4
Decidir modo de execucao
Condicao Metodo
sequential + isDAGWorkflow(steps) === true taskOrchestrator.executeDAG()
sequential + isDAGWorkflow(steps) === false taskOrchestrator.executeSequential()
step_by_step (ou qualquer outro) taskOrchestrator.executeStepByStep()
5
Tratamento de erros

Se qualquer erro ocorrer, publica evento error no canal Redis para o frontend exibir ao usuario

export async function processExecutionJob(job: Job<ExecutionJobData>): Promise<void> { const { conversationId, content, stepIndex, projectPath, workflowType, userId, attachments } = job.data const pub = getPublisher() const channel = `execution:${conversationId}` const conversation = await prisma.conversation.findUnique({ where: { id: conversationId }, include: { workflow: { include: { steps: { orderBy: { stepOrder: 'asc' } } } } }, }) if (!conversation) { await pub.publish(channel, JSON.stringify({ event: 'error', data: { message: 'Conversation not found' } })) return } try { if (workflowType === 'sequential' && isDAGWorkflow(steps)) { await taskOrchestrator.executeDAG(context, content) } else if (workflowType === 'sequential') { await taskOrchestrator.executeSequential(context, content) } else { await taskOrchestrator.executeStepByStep(context, content, currentIndex) } } catch (error) { await pub.publish(channel, JSON.stringify({ event: 'error', data: { message: error.message }, })) } }
