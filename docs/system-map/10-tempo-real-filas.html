Heartbeat

Ping a cada 15 segundos para manter conexao ativa atraves de proxies e evitar timeouts de conexao ociosa.

Formato do Evento SSE

Endpoints SSE

POST /conversations/:id/stream

Envia mensagem e recebe stream SSE. Executa o workflow completo.

GET /conversations/:id/watch

Observa execucao ativa via SSE (read-only). Reconecta apos F5 ou nova aba.

GET /conversations/:id/execution-status

Verifica se ha execucao ativa. Usado para auto-reconexao no mount.

Fases do Streaming

Ciclo de vida de uma requisicao de streaming, da conexao ao cleanup

FASE 1 0.0s

Preparacao

Frontend limpa o estado, inicia AbortController, define fase preparing.

clearStreaming()
setStreaming(true)
fetch POST /stream
phase: preparing
FASE 2 ~0.1s

Conexao

Backend estabelece conexao SSE, registra handlers de eventos, inicia heartbeat de 15s.

writeHead 200 (SSE)
Register event handlers
setInterval heartbeat
phase: connecting
FASE 3 0.1s - Ns

Execucao

AI processa, eventos fluem em tempo real. Conteudo e actions renderizados instantaneamente.

step_start events
stream (high freq)
step_complete
message_saved
phase: ai_thinking
phase: streaming
FASE 4 N+

Finalizacao

Evento complete recebido, cleanup no frontend e backend, invalidacao de queries.

event: complete
clearInterval heartbeat
off() event handlers
invalidateQueries()
setStreaming(false)

Auto-Reconexao

useEffect (mount)
GET /execution-status
active=true + state=running
watchExecution()

Ao reabrir a aba ou recarregar a pagina, o frontend detecta execucoes ativas via GET /execution-status e se reconecta automaticamente via GET /watch.

Design Hibrido

A fila existe e funciona, porem o caminho principal usa execucao inline via SSE. A fila esta disponivel para cargas elevadas ou workers dedicados.

Redis Pub/Sub

Comunicacao entre processos usando channels Redis

Padrao de Conexoes

Publisher

Conexao dedicada para publicar mensagens. Usada pelo execution-worker para notificar execucao.

Subscriber

Conexao dedicada para assinar channels. Recebe notificacoes de execucao de outros processos.

Webhook Delivery

Sistema de notificacoes externas com retry exponencial

Fluxo de Delivery

1
Trigger

Evento e emitido pelo orchestrator (execution_complete, step_complete, etc.)

dispatch(event, data, userId)
2
Busca Webhooks

Consulta o banco por webhooks registrados para este evento e usuario.

3
Cria Delivery Record

Registra no banco com status pending antes de enviar.

4
Assina Payload

HMAC-SHA256 com secret unico do webhook.

X-Webhook-Signature: sha256=abc123...
5
Fire-and-Forget

Envio assincrono. Erros sao tratados sem bloquear o fluxo principal.

Logica de Retry

Max Attempts 3
Timeout por Attempt 10 segundos
Backoff Exponencial (1s, 4s, 9s)
Status Tracking Prisma (webhookDelivery)
1 Tentativa 1 — immediate
2 Tentativa 2 — +1s
3 Tentativa 3 — +4s
X Falho apos 3 — +9s
Headers Enviados
Content-Typeapplication/json
X-Webhook-Signaturesha256=...
X-Webhook-Eventexecution_complete
