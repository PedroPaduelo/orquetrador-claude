📊 Visao Geral

O sistema de tokens e budget do Orquetrador Claude oferece controle granular sobre o consumo de tokens da API por usuario. Cada usuario possui um UserTokenBudget que define limites diarios e mensais, politica de excedente (block/warn/allow) e rastreamento de custos em USD.

O rastreamento ocorre em multiplas camadas: desde o detalhe por execucao (ExecutionTrace), passando pelo historico de uso (TokenUsageHistory), agregacao horaria (HourlyTokenUsage), metricas diarias (DailyExecutionMetrics) e totais por conversa (ExecutionAggregate).

Antes de cada execucao, o TokenBudgetService.checkBudget() verifica se o usuario tem tokens disponiveis. Apos a execucao, o uso e registrado atomicamente via $transaction para evitar race conditions.

⚠️ Overage Policies

O enum OveragePolicy define o comportamento quando o usuario excede o orcamento. Configuravel por usuario no campo overagePolicy do UserTokenBudget.

🛑
block
default

Execucao negada quando o orcamento e excedido. O checkBudget() retorna { allowed: false } e a execucao nao inicia.

// Budget check result { "allowed": false, "reason": "Daily token limit exceeded" }
⚠
warn
monitoramento

Execucao permitida mesmo com orcamento excedido, mas um warning e emitido via SSE para o frontend e registrado no log.

// Warning emitido via SSE { "type": "budget_warning", "dailyUsage": 5200000, "dailyLimit": 5000000 }
✅
allow
sem restricao

Nenhuma restricao aplicada. Util para contas admin ou ambientes de teste. O uso continua sendo rastreado normalmente.

// Always allowed { "allowed": true } // Tracking still active

🔄 Token Tracking Flow

Fluxo completo desde a execucao ate a persistencia em todas as camadas de agregacao:

⚡ Execution — Claude CLI processa a tarefa
↓
📤 Engine retorna usage — inputTokens, outputTokens, cacheTokens, costUsd
↓
📝 TokenUsageHistory.create() — Registro detalhado por execucao
↓
📊 HourlyTokenUsage.upsert() — Agregacao por hora + modelo
↓

↓
📅 DailyExecutionMetrics.upsert() — Resumo diario consolidado

⚙️ TokenBudgetService

Servico central em src/domains/execution/budget/token-budget-service.ts. Gerencia verificacao de orcamento, registro de uso e resumos, tudo em transacoes atomicas para evitar race conditions.

🚫
checkBudget(userId: string)
Retorna: Promise<BudgetCheck>

Verifica se o usuario tem tokens disponiveis. Chamado antes de cada execucao pelo motor de execucao. Auto-reseta contadores se o dia ou mes mudou.

interface BudgetCheck { allowed: boolean reason?: string } // Verifica limite diario if (budget.currentDailyUsage >= budget.dailyLimit) return { allowed: false, reason: 'Daily token limit exceeded' } // Verifica limite mensal if (budget.currentMonthlyUsage >= budget.monthlyLimit) return { allowed: false, reason: 'Monthly token limit exceeded' } return { allowed: true }
📝
recordUsage(userId, inputTokens, outputTokens)
Retorna: Promise<void>

Registra o consumo de tokens apos uma execucao. Usa $transaction para incremento atomico, garantindo que nao ha race conditions entre execucoes concorrentes.

async recordUsage(userId, inputTokens, outputTokens) { const total = inputTokens + outputTokens if (total <= 0) return await prisma.$transaction(async (tx) => { await this.getOrCreateInTx(tx, userId) // resets if needed await tx.userTokenBudget.update({ where: { userId }, data: { currentDailyUsage: { increment: total }, currentMonthlyUsage: { increment: total }, }, }) }) }
📊
getUsageSummary(userId: string)
Retorna: Promise<UsageSummary>

Retorna o resumo de uso com porcentagens calculadas. Alimenta o TokenBudgetBar no frontend e o endpoint /auth/budget.

interface UsageSummary { dailyUsage: number // ex: 1250000 dailyLimit: number // ex: 5000000 monthlyUsage: number // ex: 72000000 monthlyLimit: number // ex: 100000000 dailyPercent: number // ex: 25 monthlyPercent: number // ex: 72 }
🔧
getOrCreate(userId) / getOrCreateInTx(tx, userId)
Metodo interno — auto-reset de contadores

Busca ou cria o budget do usuario. Automaticamente reseta contadores quando detecta mudanca de dia ou mes. A versao InTx recebe o client de transacao para operar dentro de $transaction.

// Reset diario: compara dia/mes/ano if (now.getDate() !== lastDaily.getDate() || ...) { updateData.currentDailyUsage = 0 updateData.lastDailyResetAt = now } // Reset mensal: compara mes/ano if (now.getMonth() !== lastMonthly.getMonth() || ...) { updateData.currentMonthlyUsage = 0 updateData.lastMonthlyResetAt = now }

📦 Camadas de Agregacao

O rastreamento de tokens ocorre em 5 camadas com granularidades diferentes, formando uma piramide de agregacao:

🔍 ExecutionTrace — Detalhe por chamada CLI (tokens, custo, cache, duracao)
↓
📝 TokenUsageHistory — Historico por execucao (inputTokens, outputTokens, costUsd, model)
↓
⌚ HourlyTokenUsage — Agregacao horaria (hour, tokens, costUsd, model)
↓
📅 DailyExecutionMetrics — Resumo diario (total, success, p50/p95, cost)
↓
📊 ExecutionAggregate — Totais por conversa/workflow
📝
TokenUsageHistory
Tabela: token_usage_history

Registro individual por execucao. Cada chamada ao CLI gera uma entrada.

// Campos id, inputTokens, outputTokens, totalCostUsd, model, executionId, userId, createdAt // Indexes @@index([userId]) // busca por usuario @@index([createdAt]) // busca por periodo @@index([executionId])// busca por execucao
⌚
HourlyTokenUsage
Tabela: hourly_token_usage

Agregacao por hora, usuario e modelo. Usa upsert com unique constraint [hour, userId, model].

// Campos id, hour, tokens, costUsd, model, userId, createdAt // Unique constraint para upsert @@unique([hour, userId, model]) @@index([userId]) @@index([hour])
📅
DailyExecutionMetrics
Tabela: daily_execution_metrics

Resumo diario consolidado. Inclui percentis de duracao (p50/p95), breakdown por erro e modelo.

// Campos id, date, totalExecutions, successCount, failureCount, totalInputTokens, totalOutputTokens, totalCostUsd, avgDurationMs, p50DurationMs, p95DurationMs, uniqueWorkflows, uniqueUsers, errorBreakdown (Json), modelBreakdown (Json) // Unique + Index @@unique([date]) @@index([date])
📊
ExecutionAggregate
Tabela: ExecutionAggregate

Totais acumulados por execucao/conversa. Acompanha steps completados, falhos e skipped.

// Campos id, executionId (unique), conversationId, workflowId, totalSteps, completedSteps, failedSteps, skippedSteps, totalInputTokens, totalOutputTokens, totalCostUsd, totalDurationMs, startedAt, completedAt, finalStatus (ExecutionAggregateStatus: running | completed | failed | cancelled) // Indexes @@index([conversationId]) @@index([workflowId]) @@index([startedAt])
