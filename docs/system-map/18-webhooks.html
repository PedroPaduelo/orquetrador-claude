🌐 Visao Geral

Webhooks outbound notificam sistemas externos quando eventos ocorrem no orquestrador. Quando uma execucao completa, um step falha ou um workflow inicia, o sistema encontra todos os webhooks registrados para aquele evento, assina o payload com HMAC-SHA256 e faz um HTTP POST para a URL configurada. Entregas mal-sucedidas sao retentadas automaticamente com backoff exponencial.

Detalhes do Fluxo
  1. Evento ocorre — Ex: execution_completed no task-orchestrator
  2. Match webhooks — findByEvent(event, userId) busca webhooks habilitados cujos events[] incluem o evento ou *
  3. HMAC sign — signPayload(json, secret) gera sha256=<hex>
  4. Create WebhookDelivery — Status pending no banco
  5. HTTP POST — fetch(url, { body, headers }) com timeout 10s
  6. Track response — Atualiza responseCode, responseBody, status
  7. Retry on failure — Ate 3 tentativas com backoff exponencial (1s, 4s, 9s)

⚡ Eventos Disponiveis

EventoDescricaoQuando dispara
execution_completed Execucao de workflow finalizada com sucesso TaskOrchestrator marca execution como completed
step_completed Um step individual completou Flush do step com status success
step_failed Um step falhou (error/timeout) classifyError() detecta falha no step
workflow_started Workflow iniciou execucao Inicio de executeSequential/executeDAG/executeStepByStep
test Evento de teste manual POST /webhooks/:id/test
* Wildcard — recebe todos os eventos Qualquer dispatch

Wildcard: Um webhook com events: ["*"] recebe TODOS os eventos disparados. O filtro acontece no findByEvent() que verifica events.includes(event) || events.includes('*').

🔒 HMAC-SHA256 Signing

1
Processo de Assinatura
  1. Webhook tem um secret gerado com randomBytes(32).toString('hex')
  2. Secret armazenado AES-encrypted no banco via encrypt()
  3. No dispatch, secret e decriptado via decrypt()
  4. createHmac('sha256', secret).update(payload).digest('hex')
  5. Assinatura enviada no header X-Webhook-Signature: sha256=<hex>
2
Verificacao no Receptor
// Exemplo de verificacao no receptor (Node.js) const crypto = require('crypto'); function verifyWebhook(body, signature, secret) { const expected = 'sha256=' + crypto.createHmac('sha256', secret) .update(body) .digest('hex'); return crypto.timingSafeEqual( Buffer.from(signature), Buffer.from(expected) ); }

Sempre use timingSafeEqual para evitar timing attacks na comparacao de assinaturas.

▼

Retorna detalhes incluindo o secret decriptado. Valida ownership via findFirst({ where: { id, userId } }).

// Response { "id": "uuid", "url": "https://example.com/hook", "events": ["execution_completed", "step_failed"], "secret": "hex-64-chars...", "enabled": true, "createdAt": "2026-03-24T10:00:00.000Z", "updatedAt": "2026-03-24T12:00:00.000Z" }
POST /webhooks — Criar webhook
▼

Cria um novo webhook. O secret e gerado automaticamente com randomBytes(32).toString('hex'), armazenado com encrypt() e retornado em texto puro na resposta (unica vez).

// Body { "url": "https://example.com/hook", // obrigatorio, z.string().url() "events": ["execution_completed"], // obrigatorio, min 1 "enabled": true // opcional, default true } // Response (201) { "id": "uuid", "url": "https://example.com/hook", "events": ["execution_completed"], "enabled": true, "secret": "a1b2c3d4...", "createdAt": "2026-03-24T10:00:00.000Z" }
PUT /webhooks/:id — Atualizar webhook
▼

Atualiza um webhook existente. Usa $transaction com verificacao de ownership interna. O secret NAO pode ser alterado (regenere criando um novo webhook).

// Body (todos opcionais) { "url": "https://new-url.com/hook", "events": ["step_failed", "step_completed"], "enabled": false } // Response { "id", "url", "events", "enabled" }

▼

Remove o webhook e todas as suas deliveries (cascade). Usa $transaction com verificacao de ownership.

// Response: 204 No Content

▼

Retorna as ultimas 50 deliveries do webhook, ordenadas por createdAt DESC. Valida ownership antes de listar.

🔄 Retry & Backoff Exponencial

R
deliverWithRetry()

Cada delivery tem ate 3 tentativas. O backoff e quadratico: attempt * attempt * 1000ms.

TentativaDelay antesStatus se falharStatus se OK
1 Imediato pending (continua) success
2 1 segundo pending (continua) success
3 4 segundos failed (final) success

Timeout por request: 10 segundos (AbortSignal.timeout(10_000)). Response body truncado em 1000 caracteres. Fire-and-forget com .catch() para nao bloquear o dispatch.

🛡 Seguranca

✅
Ownership Validation

Todas as operacoes (GET, PUT, DELETE, deliveries, test) validam userId via findFirst({ where: { id, userId } }). Update e delete usam $transaction para evitar race conditions.

🔐
Secret Encryption

Secrets sao armazenados com AES encryption via encrypt() / decrypt() do modulo src/lib/crypto.ts. O secret em texto puro so e retornado no momento da criacao (POST /webhooks).

⏱
Timeout Protection

Cada HTTP POST tem timeout de 10 segundos via AbortSignal.timeout(10_000). Response body e truncado em 1000 chars para evitar payloads maliciosos no banco.

🔥
Fire-and-Forget

Deliveries sao executadas assincronamente com .catch(). Se o dispatch falhar, o evento original nao e afetado. Erros sao logados no console.

⇄ Inbound vs Outbound

📤
Outbound Webhooks
Este modulo

Notificam sistemas externos quando eventos ocorrem no orquestrador.

  • Dominio: src/domains/webhooks/
  • Modelo: Webhook + WebhookDelivery
  • Fluxo: Evento interno → HTTP POST → Sistema externo
  • Autenticacao: HMAC-SHA256
📥
Inbound Webhooks
WorkflowTrigger

Recebem chamadas externas para iniciar workflows no orquestrador.

  • Dominio: src/domains/triggers/
  • Modelo: WorkflowTrigger (type=webhook_inbound)
  • Fluxo: Sistema externo → POST /triggers/webhook/:id → Inicia workflow
  • Docs: Pagina 17 - Triggers
